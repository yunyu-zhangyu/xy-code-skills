# xy-cloud 避坑指南

> M0 基于项目源码扫描自行归纳 | 2026-06-12
> 只列出代码中确实存在的坑点，每个条目附带证据路径。

---

## 1. DO 继承选错：BaseDO 还是 TenantBaseDO？

**症状**：DO 继承了 TenantBaseDO 但对应的表没有 tenant_id 字段 → 插入时报 "tenant_id cannot be null"；或者 DO 继承了 BaseDO 但功能需要多租户隔离 → 数据跨租户可见。

**判断标准**：

| 场景 | 应该继承 | 证据 |
|------|---------|------|
| 需要多租户隔离的业务数据 | `TenantBaseDO` | `platform-framework/platform-spring-boot-starter-biz-tenant/src/main/java/cn/shutan/platform/framework/tenant/core/db/TenantBaseDO.java` — 追加了 `tenantId` 字段 |
| 基础配置、系统级数据（所有租户共享） | `BaseDO` | `platform-framework/platform-spring-boot-starter-mybatis/src/main/java/cn/shutan/platform/framework/mybatis/core/dataobject/BaseDO.java` — 没有 tenantId |

**项目中实际使用的例子**：
- `ConfigDO` → 继承 `BaseDO`（配置全局共享，不需要租户隔离）
- 证据：`platform-module-infra/platform-module-infra-biz/src/main/java/cn/shutan/platform/module/infra/dal/dataobject/config/ConfigDO.java`

> 不确定时看团队的数据库设计文档或问负责人。

## 2. @KeySequence 对 MySQL 多余但通常保留

**症状**：DO 上的 `@KeySequence("xxx_seq")` 让新人困惑。

**原因**：框架为了兼容 Oracle、PostgreSQL、Kingbase 等数据库，在主键上添加了 `@KeySequence`。如果项目只用 MySQL，这个注解不生效，可以不写，但现有代码通常保留它。

**证据**：`platform-module-infra/platform-module-infra-biz/src/main/java/cn/shutan/platform/module/infra/dal/dataobject/config/ConfigDO.java` 第 18 行：
```java
@KeySequence("infra_config_seq") // 用于 Oracle、PostgreSQL、Kingbase、DB2、H2 数据库的主键自增。如果是 MySQL 等数据库，可不写。
```

> 如果是纯 MySQL 项目，新建 DO 时可以省略 @KeySequence；如果已经有就保留。

## 3. 逻辑删除自动过滤（容易忘）

**症状**：写 `selectList` 发现数据不对——带 `deleted = 1` 的数据没查出来。或者调用 `deleteById` 后数据还在数据库里（只是标记为删除）。

**原因**：`BaseDO` 中的 `deleted` 字段标注了 `@TableLogic`，MyBatis-Plus 自动在所有查询加 `WHERE deleted = 0`，所有 `delete` 操作变成 `UPDATE SET deleted = 1`。

**证据**：`platform-framework/platform-spring-boot-starter-mybatis/src/main/java/cn/shutan/platform/framework/mybatis/core/dataobject/BaseDO.java` 第 31 行：
```java
@TableLogic
private Boolean deleted;
```

> 如果要查已删除的数据（如导出含已删除记录），需要在 Mapper XML 中写原生 SQL 绕过过滤。

## 4. 字段自动填充：不要手动 set createTime/updateTime

**症状**：手动调了 `setCreateTime(now)` 但值被框架覆盖。

**原因**：BaseDO 的 `createTime`、`updateTime`、`creator`、`updater` 字段标注了 `@TableField(fill = FieldFill.INSERT/INSERT_UPDATE)`，MyBatis-Plus 自动在插入/更新时填充值。

**证据**：`platform-framework/platform-spring-boot-starter-mybatis/src/main/java/cn/shutan/platform/framework/mybatis/core/dataobject/BaseDO.java` 第 17-30 行：
```java
@TableField(fill = FieldFill.INSERT)
private LocalDateTime createTime;
@TableField(fill = FieldFill.INSERT_UPDATE)
private LocalDateTime updateTime;
```

> Service 中不需要给这些字段赋值，框架会处理。

## 5. VO 字段名和 DO 字段名可能不一致（Convert 需要注意）

**症状**：RespVO 返回前端时某个字段是 null，但 DO 中有值。

**原因**：DO 字段名和 VO 字段名可能不同（如 DO 是 `configKey`，VO 是 `key`），Convert 中通过 `@Mapping` 做转换，如果忘记映射会是 null。

**证据**：`platform-module-infra/platform-module-infra-biz/src/main/java/cn/shutan/platform/module/infra/convert/config/ConfigConvert.java` 第 23 行：
```java
@Mapping(source = "configKey", target = "key")
ConfigRespVO convert(ConfigDO bean);
```

> 新建 Convert 时要检查 DO 和 VO 的字段名是否对应，不对应的地方加 @Mapping 映射。

## 6. Mapper XML 文件必须创建（即使没写 SQL）

**症状**：运行时出现 `Invalid bound statement (not found)` 异常。

**原因**：`BaseMapperX` 中的 default 方法不需要 XML，但如果你后续在 Mapper 接口中加了**非 default 方法**（如自定义 SQL 查询），MyBatis 会去找同名的 XML 文件。如果文件不存在就报错。

**证据**：`platform-module-infra/platform-module-infra-biz/src/main/resources/mapper/` 目录下的 XML 文件也都包含这样一个注释块（来自代码生成器的规范）：

```xml
<!--
    一般情况下，尽可能使用 Mapper 进行 CRUD 增删改查即可。
    无法满足的场景，例如说多表关联查询，才使用 XML 编写 SQL。
-->
```

> 每个 Mapper 都应该有一个同名的空 XML 文件放在 resources/mapper/ 下，避免后续加方法时报错。

## 7. @PreAuthorize 权限表达式格式

**症状**：接口调用返回 403 Forbidden。

**原因**：`@PreAuthorize("@ss.hasPermission('{module}:{resource}:{action}')")` 中的参数需要和数据库中的权限标识完全匹配。

**证据**：`platform-module-infra/platform-module-infra-biz/src/main/java/cn/shutan/platform/module/infra/controller/admin/config/ConfigController.java` 第 43 行：
```java
@PreAuthorize("@ss.hasPermission('infra:config:create')")
```

格式固定为 `{模块前缀}:{资源名}:{操作}`：
- 模块前缀：module 模块对应 `infra`、system 对应 `system`
- 资源名：通常和 URL 路径第二段一致
- 操作：create / update / delete / query / export

> 新建 Controller 时，@PreAuthorize 的权限标识要和前端菜单配置一致，否则有权限的用户也会被拒绝。

## 8. eos-demand-response 的 biz 模块名是 -diz 不是 -biz

**症状**：`mvn compile -pl platform-module-eos-demand-response/platform-module-eos-demand-response-biz` 报找不到模块。

**原因**：eos-demand-response 的 biz 子模块被命名为 `platform-module-eos-demand-response-diz`（不是 biz）。

> 编译和执行测试时确认模块名正确：`platform-module-eos-demand-response/platform-module-eos-demand-response-diz`。
