## 1. 模块分层

### 1.1 标准分层结构

```
platform-module-xxx
├── xxx-api (轻量 JAR)
│   ├── api/          Feign 客户端
│   ├── api/xxx/dto/  响应 DTO
│   ├── api/xxx/vo/   请求 VO
│   └── enums/        常量/枚举
└── xxx-biz (可执行 JAR)
    ├── controller/admin/  管理后台 API
    ├── controller/app/    APP 端 API
    ├── service/           Service 接口 + Impl
    ├── dal/dataobject/    DO 实体
    ├── dal/mysql/         MyBatis-Plus Mapper
    ├── convert/           MapStruct 转换器
    ├── api/               Feign 实现
    ├── config/            配置类
    ├── framework/         模块级框架配置
    ├── job/               XXL-Job
    ├── mq/                消息队列
    └── enums/             模块枚举
```

### 1.2 各层职责说明

| 层 | 职责 | 约束 |
|----|------|------|
| **Controller** | 接收 HTTP 请求，参数校验，调用 Service | 不包含业务逻辑，不直接操作 DAO |
| **Service** | 核心业务逻辑，事务管理 | 接口与实现分离，Impl 标注 `@Service` |
| **DAO (dal)** | 数据库 CRUD 操作 | Mapper 接口 + DO 实体 + XML 文件 |
| **Convert** | 对象转换（DO ↔ VO ↔ DTO） | 使用 MapStruct 或 `BeanUtils.toBean()` |
| **api 模块** | Feign 接口定义 + 常量枚举 | 不含业务逻辑，作为轻量 JAR 被其他模块依赖 |

| 层级 | 格式 | 示例 |
|------|------|------|
| 控制器 | `controller.admin.{domain}` | `controller.admin.dept` |
| 控制器 VO | `controller.admin.{domain}.vo` | `controller.admin.dept.vo` |
| Service | `service.{domain}` | `service.dept` |
| DO | `dal.dataobject.{domain}` | `dal.dataobject.dept` |
| Mapper | `dal.mysql.{domain}` | `dal.mysql.dept` |

- Controller: `cn.shutan.platform.module.eos.controller.admin.energyprice.EnergyPriceController`
- Service: `cn.shutan.platform.module.eos.service.energyprice.EnergyPriceService`

## 3. Controller 规范

### 3.1 命名与路径约定

```
HTTP  方法名            路径         返回
POST  create{Entity}    "create"     CommonResult<Long>
PUT   update{Entity}    "update"     CommonResult<Boolean>
DELETE delete{Entity}   "delete"     CommonResult<Boolean>
GET   get{Entity}       "/get"       CommonResult<RespVO>
GET   get{Entity}List   "/list"      CommonResult<List<RespVO>>
GET   get{Entity}Page   "/page"      CommonResult<PageResult<RespVO>>
```

### 3.2 类注解

```java
@Tag(name = "管理后台 - XXX")                 // Swagger 分组
@RequestMapping("/{module}/{resource}")      // 请求路径
@PreAuthorize("@ss.hasPermission('{module}:{resource}:{action}')")  // 权限控制
```

### 3.3 @Parameter 使用时机

Controller 方法参数应使用 `@Parameter` 注解为 Swagger 文档补充字段说明：

```java
@GetMapping("/page")
public CommonResult<PageResult<XxxRespVO>> getXxxPage(
    @Parameter(description = "分页参数") XxxPageReqVO pageReqVO) {
    // ...
}
```

### 3.4 @Valid 使用规则

- 请求 VO 参数必须标注 `@Valid` 以触发 JSR-303 校验
- 类上必须标注 `@Validated` 以使 `@Valid` 生效
- 校验失败统一由 `GlobalExceptionHandler` 返回 400

```java
@RestController
@RequestMapping("/admin/energyprice")
@Validated
@Tag(name = "管理后台 - 能源价格")
public class EnergyPriceController {

    @PostMapping("/create")
    @PreAuthorize("@ss.hasPermission('eos:energyprice:create')")
    public CommonResult<Long> createEnergyPrice(
            @Valid @RequestBody EnergyPriceSaveReqVO vo) {
        return CommonResult.success(energyPriceService.createEnergyPrice(vo));
    }
}
```

## 4. Service 层

### 4.1 命名规范

- 接口: `{Entity}Service.java` → `DeptService.java`
- 实现: `{Entity}ServiceImpl.java` → `DeptServiceImpl.java`
- 方法: `create{Entity}` / `update{Entity}` / `delete{Entity}` / `get{Entity}` / `get{Entity}List` / `get{Entity}Page`

### 4.2 @Transactional 使用规则

- 写操作（create/update/delete）方法必须标注 `@Transactional(rollbackFor = Exception.class)`
- 读操作不加事务注解，由 MyBatis 自行管理
- 默认回滚策略: Spring 默认只对 `RuntimeException` 和 `Error` 回滚，必须显式指定 `rollbackFor = Exception.class`

**常见错误**：
- 在 private 方法上加 `@Transactional`（AOP 代理不生效）
- 同类方法调用绕过代理（`this.xxx()` 不触发事务）
- 在多线程/异步方法中使用事务（`@Async` + 事务 = 线程本地事务失效）

## 5. VO 命名

| VO | 用途 | 位置 |
|----|------|------|
| `{Entity}SaveReqVO` | 创建/更新请求 | controller/admin/{domain}/vo |
| `{Entity}PageReqVO` | 分页查询请求 | controller/admin/{domain}/vo (继承 PageParam) |
| `{Entity}ListReqVO` | 列表查询请求 | controller/admin/{domain}/vo |
| `{Entity}RespVO` | 响应 VO | controller/admin/{domain}/vo |

### 5.1 VO 字段和注解规范

```java
public class EnergyPriceSaveReqVO {

    @Schema(description = "价格名称", required = true, example = "峰时电价")
    @NotBlank(message = "价格名称不能为空")
    private String name;

    @Schema(description = "价格值（元/kWh）", required = true, example = "1.05")
    @NotNull(message = "价格值不能为空")
    @DecimalMin(value = "0", message = "价格值必须大于等于 0")
    private BigDecimal price;

    @Schema(description = "状态", required = true, example = "1")
    @NotNull(message = "状态不能为空")
    private Integer status;
}
```

- **@Schema**: Swagger 文档注解，`required` 标记必填，`example` 提供示例值
- **@NotBlank**: 字符串非空校验
- **@NotNull**: 对象/数值非空校验
- **@DecimalMin/@DecimalMax**: 数值范围校验

## 6. 对象转换

- DO → VO: `BeanUtils.toBean(xxxDO, XxxRespVO.class)`
- DO 列表 → VO 列表: `BeanUtils.toBean(xxxDOList, XxxRespVO.class)`
- 分页转换: `BeanUtils.toBean(pageResult, XxxRespVO.class)`
- 带补充字段: `BeanUtils.toBean(xxxDO, XxxRespVO.class, vo -> vo.setXxx(...))`

`BeanUtils.toBean()` 底层基于 Cglib BeanCopier，比 PropertyUtils 快约 400%，适合批量转换场景。MapStruct 适合编译期生成转换代码的场景，二者择一即可，不要混用。

## 7. API 防护

```java
@Idempotent(timeout = 1, message = "请勿重复提交")
@RateLimiter(count = 100, time = 60)
```

- **@Idempotent**: 幂等注解，基于 Redis 实现，防止接口在 timeout 秒内重复提交
- **@RateLimiter**: 限流注解，限制每 time 秒内最多 count 次请求

## 9. Mapper XML 规范

### 9.1 文件位置

`src/main/resources/mapper/{Entity}Mapper.xml`（resources 目录平铺，无子目录）

> **必须生成**：即使当前 Mapper 接口只有 default 方法（无需 XML），也必须创建 Mapper XML 文件。这确保后续添加非 default 方法（如自定义 SQL）时不会因缺少 XML 而触发 `Invalid bound statement (not found)` 异常。

### 9.2 文件结构

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN"
    "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="cn.shutan.platform.module.{module}.dal.mysql.{domain}.{Entity}Mapper">
    <!-- 规范注释块 -->
</mapper>
```

### 9.3 规范注释块

每个 Mapper XML 文件开头必须包含以下注释块：

```xml
<!--
    一般情况下，尽可能使用 Mapper 进行 CRUD 增删改查即可。
    无法满足的场景，例如说多表关联查询，才使用 XML 编写 SQL。
    代码生成器暂时只生成 Mapper XML 文件本身，更多推荐 MybatisX 快速开发插件来生成查询。
    文档可见：https://www.shutan.cn/MyBatis/x-plugins/
 -->
```

### 9.4 SQL 编写规范

- 使用 `resultType`（非 resultMap），值为全限定类名
- 不使用 `<sql>` 片段
- 不使用 `<insert>`/`<update>`/`<delete>`（CRUD 由 BaseMapperX 处理）
- 动态 SQL 使用 `<where>` + `<if test="...">`
- 所有查询含逻辑删除表需加 `AND deleted = 0`
- 比较操作符使用 CDATA：`<![CDATA[<=]]>`
- 返回值类型：DO 用 dataobject 全路径，VO 用 controller/admin 全路径

## 10. DO 编码规范

### 10.1 DO 注解全集

```java
@TableName("eos_energy_price")
@Data
@Builder
@NoArgsConstructor
@AllArgsConstructor
@EqualsAndHashCode(callSuper = true)
public class EnergyPriceDO extends TenantBaseDO {

    @TableId(type = IdType.AUTO)
    private Long id;

    private String name;

    @TableField(typeHandler = JacksonTypeHandler.class)
    private List<String> tags;  // JSON 字段

    private BigDecimal price;

    private Integer status;

    @TableField(exist = false)
    private String extraField;  // 非数据库字段
}
```

| 注解 | 用途 | 备注 |
|------|------|------|
| `@TableName` | 指定表名 | 防止 MyBatis-Plus 自动生成错误表名 |
| `@TableId(type = IdType.AUTO)` | 自增主键 | 非自增表使用 `IdType.ASSIGN_ID`（雪花） |
| `@TableField(typeHandler = ...)` | JSON 字段类型处理器 | 自动在 Java 对象与 JSON 字符串间转换 |
| `@TableField(exist = false)` | 标记非数据库字段 | 查询和插入时忽略 |
| `@Data` + `@Builder` | Lombok 组合 | Builder 模式需搭配 `@NoArgsConstructor` + `@AllArgsConstructor` |

### 10.2 autoResultMap 规则

当 DO 中包含 `@TableField(typeHandler = JacksonTypeHandler.class)` 注解的 JSON 字段时，`@TableName` 必须指定 `autoResultMap = true`：

```java
@TableName(value = "eos_energy_price", autoResultMap = true)
```

否则查询时 JSON 字段将无法正确反序列化，表现为该字段为 null 或查询报错。

### 10.3 @Builder 模式

Lombok `@Builder` 在使用继承时有约束：

- 子类和父类都必须有 `@NoArgsConstructor` + `@AllArgsConstructor`
- 不能直接通过 builder 链设置父类字段（如 `tenantId`），需要构造后 `setXxx()`

## 11. DO 继承选择

### 11.1 父类决策树

```
DO 表是否包含 tenant_id 字段？
├── 是 → 继承 TenantBaseDO
└── 否 → 继承 BaseDO
```

### 11.2 两个父类对比

| | BaseDO | TenantBaseDO |
|--|--------|-------------|
| **类路径** | `cn.shutan.platform.framework.mybatis.core.dataobject.BaseDO` | `cn.shutan.platform.framework.mybatis.core.dataobject.TenantBaseDO` |
| **包含字段** | createTime, updateTime, creator, updater, deleted | BaseDO 全部字段 + tenantId |
| **租户过滤** | 不触发 | 触发 |
| **适用场景** | 跨租户共享表、eos 模块 | 多租户数据表 |

### 11.3 选错后果

> ⚠️ **这是静默失败**：继承 `BaseDO` 而非 `TenantBaseDO` 时，`TenantDatabaseInterceptor` 不会拦截 SQL，**数据将在租户间泄露**，且不会产生任何异常或错误日志。

### 11.4 各模块对照表

| 模块 | 推荐父类 | 原因 |
|------|---------|------|
| `platform-module-system` | `TenantBaseDO` | 多租户数据，需要租户隔离 |
| `platform-module-infra` | `TenantBaseDO` / `BaseDO` | 部分表跨租户共享，按实际情况选择 |
| `platform-module-eos` | `BaseDO` | eos 模块通常不涉及多租户 |

## 12. 字段类型规范

### 12.1 Java ↔ MySQL 类型映射

| Java 类型 | MySQL 类型 | 说明 |
|-----------|-----------|------|
| `Long` | `BIGINT` | 主键、外键、租户 ID |
| `Integer` | `INT` / `TINYINT` | 状态、枚举值 |
| `String` | `VARCHAR` / `TEXT` | 名称、描述、URL |
| `BigDecimal` | `DECIMAL` | 金额、价格 |
| `LocalDateTime` | `DATETIME` | 时间字段 |
| `Boolean` | `TINYINT(1)` | 逻辑标志 |
| `List<String>` | `JSON` / `TEXT` | JSON 字段（配合 JacksonTypeHandler） |

### 12.2 状态字段分模块规则

| 模块 | status 字段类型 | 原因 |
|------|----------------|------|
| `system` | `Integer` | 使用 `CommonStatusEnum`（0/1），与框架测试工具默认行为兼容 |
| `infra` | `Integer` | 同上 |
| `eos` | `String` 或 `Integer` | 业务状态值可能是字符串编码，需结合实际设计 |

> **建议**: system/infra 模块新增实体的 `status` 字段统一使用 `Integer` 类型，确保与 `randomPojo()` 和 `randomCommonStatus()` 的默认 `Integer` 生成行为一致。

### 12.3 常见踩坑

- **时间字段用 Date**: 应使用 `LocalDateTime`，Java 8+ 的 Date API 更安全
- **金额用 Double/Float**: 浮点运算会有精度损失，必须用 `BigDecimal`
- **JSON 字段不加 typeHandler**: 存储为 JSON 的字段需要 `JacksonTypeHandler`，否则序列化/反序列化会出错
- **主键用基本类型 long**: 任何时候都用包装类型 `Long`，基本类型无法表示 null

## 13. 常见陷阱

### 陷阱 1: Mapper XML 缺失

**现象**: 运行时报 `BindingException: Invalid bound statement (not found)`  
**原因**: Mapper 接口有非 default 方法，但 XML 中缺少对应 statement  
**排查**: 对比 Mapper 接口方法名与 XML 中的 `<select>`/`<insert>`/`<update>`/`<delete>` id 是否一一对应  
**预防**: 新 Mapper 同时创建 XML 文件，即使暂时全是 default 方法

### 陷阱 2: BaseDO 当 TenantBaseDO 用

**现象**: 多租户表中租户数据互相可见  
**原因**: DO 继承了 `BaseDO` 而非 `TenantBaseDO`，`TenantDatabaseInterceptor` 不触发  
**排查**: 全局搜索 `extends BaseDO`，交叉比对表是否包含 `tenant_id` 列  
**预防**: 新建 DO 时参考 DO 继承选择决策树（第 11 节）

### 陷阱 3: JSON 字段未设置 autoResultMap

**现象**: JSON 字段查询结果为 null 或报 JSON 反序列化错误  
**原因**: `@TableName` 缺少 `autoResultMap = true`  
**排查**: 检查有 `JacksonTypeHandler` 注解字段的 DO 的 `@TableName` 属性  
**预防**: 只要 DO 中有 JSON 字段就立即加上 `autoResultMap = true`

### 陷阱 4: status 字段类型不匹配

**现象**: 测试中 `randomPojo()` 生成的 DO 的 status 字段为 null  
**原因**: `randomCommonStatus()` 返回 `Integer`，但被测 DO 的 status 是 `String` 类型，Podam 跳过该字段  
**排查**: 检查 DO 中 status 字段的 Java 类型  
**预防**: system/infra 模块新增实体统一使用 `Integer` 类型 status
