## 1. 错误码体系

### 1.1 ErrorCode 基础类

**类路径**: `cn.shutan.platform.framework.common.exception.ErrorCode`

错误码在 xy-cloud 中被封装为对象（而非单纯的数值或枚举），原因是为未来的国际化做准备（参见源码注释: `TODO 错误码设计成对象的原因，为未来的 i18 国际化做准备`）。

```java
public class ErrorCode {
    private final Integer code;   // 错误码编号
    private final String msg;     // 错误消息
}
```

关键设计决策:

- `code` 和 `msg` 均为 `final` 字段，对象一旦创建不可修改，保证线程安全。
- 全局错误码占用 `[0, 999]` 区间（参见 `GlobalErrorCodeConstants`）。
- 业务异常错误码占用 `[1 000 000 000, +∞)` 区间（参见 `ServiceErrorCodeRange`）。

### 1.2 错误码分段约定（10 位四段结构）

**类路径**: `cn.shutan.platform.framework.common.exception.enums.ServiceErrorCodeRange`

错误码使用 **10 位十进制数**，逻辑上分为四段，每段用下划线分隔以提升可读性：

| 段位 | 长度 | 含义 | 示例值 |
|------|------|------|--------|
| 第一段 | 1 位 | 类型标识（`1` = 业务异常） | `1` |
| 第二段 | 3 位 | 系统编号（如 `002` 表示 system 系统） | `002` |
| 第三段 | 3 位 | 模块编号（如 `003` 表示用户模块） | `003` |
| 第四段 | 3 位 | 具体错误码（`000~999`，模块内自增） | `000` |

完整格式示例: `1_002_003_000`

```
 1  002  003  000
 │   │    │    └─ 第四段：具体错误（第 0 号错误）
 │   │    └────── 第三段：模块编号（用户模块）
 │   └─────────── 第二段：系统编号（system 系统）
 └─────────────── 第一段：类型（1 = 业务异常）
```

**说明**: 第一段目前仅 `1`（业务异常）和 `2`（eos 模块扩展使用）。后续若新增类型可扩展，但应保持向后兼容。

### 1.4 全局错误码

**类路径**: `cn.shutan.platform.framework.common.exception.enums.GlobalErrorCodeConstants`

全局错误码占用 `[0, 999]` 区间，主要复用 HTTP 状态码语义，但使用 `0` 表示成功而非 `200`。

```java
public interface GlobalErrorCodeConstants {

    // ─── 成功 ───
    ErrorCode SUCCESS = new ErrorCode(0, "成功");

    // ─── 客户端错误（4xx） ───
    ErrorCode BAD_REQUEST          = new ErrorCode(400, "请求参数不正确");
    ErrorCode UNAUTHORIZED         = new ErrorCode(401, "账号未登录");
    ErrorCode FORBIDDEN            = new ErrorCode(403, "没有该操作权限");
    ErrorCode NOT_FOUND            = new ErrorCode(404, "请求未找到");
    ErrorCode METHOD_NOT_ALLOWED   = new ErrorCode(405, "请求方法不正确");
    ErrorCode LOCKED               = new ErrorCode(423, "请求失败，请稍后重试");
    ErrorCode TOO_MANY_REQUESTS    = new ErrorCode(429, "请求过于频繁，请稍后重试");

    // ─── 服务端错误（5xx） ───
    ErrorCode INTERNAL_SERVER_ERROR = new ErrorCode(500, "系统异常");
    ErrorCode NOT_IMPLEMENTED       = new ErrorCode(501, "功能未实现/未开启");
    ErrorCode ERROR_CONFIGURATION   = new ErrorCode(502, "错误的配置项");

    // ─── 自定义错误（900+） ───
    ErrorCode REPEATED_REQUESTS     = new ErrorCode(900, "重复请求，请稍后重试");
    ErrorCode DEMO_DENY             = new ErrorCode(901, "演示模式，禁止写操作");
    ErrorCode UNKNOWN               = new ErrorCode(999, "未知错误");
}
```

**说明**:

- `0` = 成功，使用 `CommonResult.isSuccess()` 通过 `Objects.equals(code, 0)` 判断。
- `400` = 参数校验失败，适用于 `@Validated` 校验不通过、参数缺失、参数类型不匹配等场景。
- `403` = 权限不足，由 Spring Security `AccessDeniedException` 触发。
- `500` = 系统异常兜底，此时会自动异步记录 API 错误日志。
- `900` = 重复请求（防止幂等场景下的重复提交）。
- `901` = 演示模式，禁止写操作。
- `501` = 功能未实现/未开启，用于表不存在时的友好提示（见表不存在处理机制）。

### 1.5 ErrorCodeConstants 定义模式

各模块通过 `interface` 定义错误码常量，使用 Java 下划线分隔数字字面量提升可读性:

```java
// 文件名: platform-module-system-api/.../enums/ErrorCodeConstants.java
public interface ErrorCodeConstants {

    // ========== AUTH 模块: 1-002-000-000 段 ==========
    ErrorCode AUTH_LOGIN_BAD_CREDENTIALS   = new ErrorCode(1_002_000_000, "登录失败，账号密码不正确");
    ErrorCode AUTH_LOGIN_USER_DISABLED     = new ErrorCode(1_002_000_001, "登录失败，账号被禁用");
    ErrorCode AUTH_LOGIN_CAPTCHA_CODE_ERROR = new ErrorCode(1_002_000_004, "验证码不正确，原因：{}");

    // ========== 用户模块: 1-002-003-000 段 ==========
    ErrorCode USER_USERNAME_EXISTS         = new ErrorCode(1_002_003_000, "用户账号已经存在");
    ErrorCode USER_MOBILE_EXISTS           = new ErrorCode(1_002_003_001, "手机号已经存在");
    ErrorCode USER_NOT_EXISTS              = new ErrorCode(1_002_003_003, "用户不存在");
    ErrorCode USER_PASSWORD_FAILED         = new ErrorCode(1_002_003_005, "用户密码校验失败");
}
```

**说明**:

- 使用 `interface` 而非 `class` 或 `enum`，因为接口中的字段隐式为 `public static final`，天然适合定义常量池。
- Java 7+ 支持数字字面量下划线 `1_002_000_000`，实际值为 `1002000000`。
- 消息模板支持 `{}` 占位符（参见 3. ServiceExceptionUtil 的格式化机制）。
- 错误码按子模块分段，段与段之间建议预留空间（如 `1-002-003-000` 到 `1-002-004-000` 之间有约 1000 个位置供单个子模块使用）。

## 2. 异常类

### 2.1 ServiceException（业务异常）

**类路径**: `cn.shutan.platform.framework.common.exception.ServiceException`

```java
@Data
@EqualsAndHashCode(callSuper = true)
public final class ServiceException extends RuntimeException {

    private Integer code;     // 业务错误码
    private String message;   // 错误提示
}
```

**设计要点**:

| 特性 | 说明 |
|------|------|
| 继承关系 | 继承 `RuntimeException`，为 **unchecked** 异常，无需显式 try-catch |
| 类修饰符 | `final` 类，**禁止继承**，确保异常类型体系清晰，避免子类泛滥 |
| 无参构造 | 提供空构造 `ServiceException()`，**避免反序列化问题**（如 Dubbo/Feign 调用时序列化框架需要无参构造） |
| 链式 setter | `setCode()` / `setMessage()` 返回 `ServiceException` 自身，支持链式调用 |

**使用方式**:

```java
// 方式一: 通过 ErrorCode 对象构造
throw new ServiceException(USER_NOT_EXISTS);           // code=1002003003, message="用户不存在"

// 方式二: 直接指定 code + message（通常由 ServiceExceptionUtil 内部使用）
throw new ServiceException(1_002_003_003, "用户不存在");
```

### 2.2 ServerException（服务器异常）

**类路径**: `cn.shutan.platform.framework.common.exception.ServerException`

```java
@Data
@EqualsAndHashCode(callSuper = true)
public final class ServerException extends RuntimeException {

    private Integer code;     // 全局错误码
    private String message;   // 错误提示
}
```

**与 ServiceException 的对比**:

| 维度 | ServiceException | ServerException |
|------|-----------------|-----------------|
| 语义 | **业务逻辑异常**，如用户不存在、密码错误 | **服务端系统异常**，如配置错误、功能未开启 |
| 错误码来源 | `ServiceErrorCodeRange` 范围的业务错误码 | `GlobalErrorCodeConstants` 范围的全局错误码 |
| 使用场景 | Service 层业务校验 | 服务端内部状态异常 |

**重要说明**: 两个类的结构完全一致（code + message 字段、final 类、无参构造、链式 setter）。区别纯粹在于**语义层面**——`ServiceException` 代表"业务说 No"，`ServerException` 代表"系统出问题了"。实际项目中 `ServiceException` 的使用频率远高于 `ServerException`。

## 3. ServiceExceptionUtil（异常工具类）

**类路径**: `cn.shutan.platform.framework.common.exception.util.ServiceExceptionUtil`

### 3.1 核心方法

```java
// 根据 ErrorCode 构造 ServiceException
public static ServiceException exception(ErrorCode errorCode)

// 根据 ErrorCode + 参数（替换 {} 占位符）
public static ServiceException exception(ErrorCode errorCode, Object... params)

// 快捷构造 400 参数校验异常
public static ServiceException invalidParamException(String messagePattern, Object... params)
```

### 3.2 使用规范

**项目约定: 通过静态导入使用 `ServiceExceptionUtil.exception()`，禁止直接 `new ServiceException()`。**

```java
import static cn.shutan.platform.framework.common.exception.util.ServiceExceptionUtil.exception;

// 无参数 —— 错误消息直接使用 ErrorCode 中的 msg
throw exception(USER_NOT_EXISTS);

// 带参数 —— 将 params 替换到 ErrorCode 消息的 {} 占位符中
throw exception(ROLE_NAME_DUPLICATE, role.getName());
// 假设 ROLE_NAME_DUPLICATE 的 msg = "已经存在名为【{}】的角色"
// 实际 message 会被格式化为: "已经存在名为【超级管理员】的角色"

// 参数校验异常
throw invalidParamException("用户ID不能为空");
```

### 3.3 {} 占位符格式化机制

工具类采用手动解析 `{}` 占位符而非 `String.format()`，原因见源码注释中的说明:

> 考虑到 String.format 在参数不正确时会报错，因此使用 `{}` 作为占位符

```java
// 底层实现逻辑
public static String doFormat(int code, String messagePattern, Object... params) {
    StringBuilder sbuf = new StringBuilder(messagePattern.length() + 50);
    int i = 0;
    for (int l = 0; l < params.length; l++) {
        int j = messagePattern.indexOf("{}", i);
        if (j == -1) {
            // 参数过多: 日志告警但不抛异常
            log.error("[doFormat][参数过多：错误码({})|错误内容({})|参数({})", ...);
            return sbuf.length() > 0 ? sbuf.toString() : messagePattern;
        }
        sbuf.append(messagePattern, i, j).append(params[l]);
        i = j + 2;
    }
    if (messagePattern.indexOf("{}", i) != -1) {
        // 参数过少: 日志告警但不抛异常，保留未替换的 {}
        log.error("[doFormat][参数过少：错误码({})|错误内容({})|参数({})", ...);
    }
    sbuf.append(messagePattern.substring(i));
    return sbuf.toString();
}
```

**设计优势**: 参数数量与占位符数量不匹配时，仅日志告警而非抛出 `IllegalFormatException`，保证业务逻辑不因格式化问题而中断。

## 4. 全局异常处理

### 4.1 GlobalExceptionHandler（Spring MVC）

**类路径**: `cn.shutan.platform.framework.web.core.handler.GlobalExceptionHandler`

标注 `@RestControllerAdvice`，将各类异常统一转换为 `CommonResult` 返回。下表展示了各类异常的处理策略:

| 异常类型 | HTTP 状态码 | 响应 msg | 日志级别 |
|----------|-------------|----------|----------|
| `MissingServletRequestParameterException` | 400 | `"请求参数缺失:{name}"` | warn |
| `MethodArgumentTypeMismatchException` | 400 | `"请求参数类型错误:{msg}"` | warn |
| `MethodArgumentNotValidException` | 400 | `"请求参数不正确:{fieldError}"` | warn |
| `BindException` | 400 | `"请求参数不正确:{fieldError}"` | warn |
| `HttpMessageNotReadableException` -> `InvalidFormatException` | 400 | `"请求参数类型错误:{value}"` | warn |
| `HttpMessageNotReadableException` -> 其他 | 500 | 兜底处理 | error |
| `ConstraintViolationException` | 400 | 第一条约束违反消息 | warn |
| `ValidationException` | 400 | 固定消息 "请求参数不正确" | warn |
| `NoHandlerFoundException` | 404 | `"请求地址不存在:{url}"` | warn |
| `HttpRequestMethodNotSupportedException` | 405 | `"请求方法不正确:{msg}"` | warn |
| `BindingException` / `Invalid bound statement (not found)` | 500 | "系统配置异常" | error |
| **ServiceException** | **业务错误码** | **ex.getMessage()** | **warn（仅首行栈）** |
| `AccessDeniedException` | 403 | "没有该操作权限" | warn |
| `Exception`（兜底） | 500 | "系统异常" | **error（异步记录）** |

> **说明**: `BindingException` 通常原因是 Mapper 接口中存在非 default 方法，但对应的 Mapper XML 文件中缺失该方法的 statement 定义。MyBatis-Plus 在启动时不检查 XML 与接口的一致性，只有在运行时调用该方法时才会抛出此异常。排查方法：对比 Mapper 接口方法名与 XML 中的 `<select>`/`<insert>`/`<update>`/`<delete>` id 是否一一对应。

#### ServiceException 的特殊处理

```java
@ExceptionHandler(value = ServiceException.class)
public CommonResult<?> serviceExceptionHandler(ServiceException ex) {
    // 忽略白名单消息（如 "无效的刷新令牌"），避免无意义的日志输出
    if (!IGNORE_ERROR_MESSAGES.contains(ex.getMessage())) {
        // 仅打印第一层 StackTraceElement，使用 warn 级别输出
        StackTraceElement[] stackTraces = ex.getStackTrace();
        for (StackTraceElement stackTrace : stackTraces) {
            if (ObjUtil.notEqual(stackTrace.getClassName(), ServiceExceptionUtil.class.getName())) {
                log.warn("[serviceExceptionHandler]\n\t{}", stackTrace);
                break;
            }
        }
    }
    return CommonResult.error(ex.getCode(), ex.getMessage());
}
```

关键行为:

1. **消息白名单过滤**: 定义在 `IGNORE_ERROR_MESSAGES` 集合中的消息（如 `"无效的刷新令牌"`），不打印任何日志，避免频繁的无效日志输出。
2. **仅首行栈**: 打印堆栈时跳过 `ServiceExceptionUtil` 内部调用栈，只取业务代码中的第一行调用栈，降低日志噪音。
3. **直接返回错误码**: HTTP 状态码为 `200`，真正的业务错误码通过 `CommonResult.code` 返回，前端通过 `code != 0` 判断业务失败。

#### 兜底异常处理

```java
@ExceptionHandler(value = Exception.class)
public CommonResult<?> defaultExceptionHandler(HttpServletRequest req, Throwable ex) {
    // 检查是否为"表不存在"的特殊情况（模块化部署时某些表未导入）
    CommonResult<?> tableNotExistsResult = handleTableNotExists(ex);
    if (tableNotExistsResult != null) {
        return tableNotExistsResult;
    }
    // 记录 error 日志 + 异步插入 API 错误日志
    log.error("[defaultExceptionHandler]", ex);
    createExceptionLog(req, ex);
    return CommonResult.error(INTERNAL_SERVER_ERROR.getCode(), INTERNAL_SERVER_ERROR.getMsg());
}
```

**表不存在特殊处理**: `handleTableNotExists()` 通过检查异常消息中是否包含表名前缀（如 `report_`、`bpm_`、`mp_`、`product_` 等），返回友好的 `501` 错误码引导用户参考文档开启对应模块。这是 xy-cloud 模块化部署架构下的特殊设计——未部署的模块不会创建对应数据库表。

#### 异步错误日志记录

兜底处理中通过 `apiErrorLogApi.createApiErrorLogAsync()` 记录完整的 API 错误日志,包含:

- 用户信息: `userId`、`userType`
- 异常信息: 异常类名、异常消息、根因消息、完整堆栈
- 代码位置: 异常抛出的类名、文件名、方法名、行号
- 请求信息: TraceId、应用名、请求 URL、请求参数（query + body）、请求方法、UserAgent、客户端 IP
- 时间戳: `exceptionTime`

#### 对外提供的静态方法

`allExceptionHandler(request, throwable)` 被声明为 `public` 的实例方法，供 Filter 等非 Spring MVC 流程调用，统一异常处理行为。

### 4.2 Gateway 全局异常处理

**类路径**: `platform-gateway/.../handler/GlobalExceptionHandler`

实现 `ErrorWebExceptionHandler`，`@Order(-1)` 确保优先级高于 Spring Cloud Gateway 默认的 `ErrorWebExceptionHandler`:

```java
@Component
@Order(-1)
public class GlobalExceptionHandler implements ErrorWebExceptionHandler {

    @Override
    public Mono<Void> handle(ServerWebExchange exchange, Throwable ex) {
        // 已 commit 的响应直接回传异常
        if (response.isCommitted()) {
            return Mono.error(ex);
        }
        CommonResult<?> result;
        if (ex instanceof ResponseStatusException) {
            result = responseStatusExceptionHandler(exchange, (ResponseStatusException) ex);
        } else {
            result = defaultExceptionHandler(exchange, ex);
        }
        return WebFrameworkUtils.writeJSON(exchange, result);
    }
}
```

**处理逻辑**:

- `ResponseStatusException`: 提取 `getRawStatusCode()` 作为错误码，`getReason()` 作为消息（TODO: 会进一步精细化翻译）。
- 其他异常: 500 兜底，记录 error 日志（TODO: 考虑是否插入异常日志）。

## 5. 事务规范 @Transactional

### 5.1 使用位置

**约定: 仅在 Service 层使用，绝不在 Controller 层使用事务。**

原因:

- Controller 负责请求路由和参数校验,不应包含业务逻辑。
- Service 层是业务逻辑的容器，事务应围绕业务操作展开。
- Controller 层开启事务会导致连接池资源过早占用，降低并发能力。

### 5.2 标准配置

```java
@Transactional(rollbackFor = Exception.class)
```

**`rollbackFor = Exception.class` 是项目标准配置**，其必要性在于:

| 配置 | 回滚行为 | 说明 |
|------|---------|------|
| 默认（无 rollbackFor） | 仅回滚 `RuntimeException` + `Error` | checked 异常（如 `IOException`）不会触发回滚 |
| `rollbackFor = Exception.class` | 回滚 `Exception` 的所有子类 | 包括 checked 异常，保证数据一致性 |

`propagation` 未自定义，默认使用 `Propagation.REQUIRED`（支持当前事务，不存在则创建新事务）。

### 5.3 典型模式

```java
@Service
public class XxxServiceImpl implements XxxService {

    @Override
    @Transactional(rollbackFor = Exception.class)
    public Long create(XxxSaveReqVO reqVO) {
        // 1. 参数校验 —— @Validated 在接口层完成
        // 2. 业务校验 —— 使用 ServiceExceptionUtil 抛异常
        if (xxxMapper.selectByName(reqVO.getName()) != null) {
            throw exception(XXX_NAME_DUPLICATE, reqVO.getName());
        }
        // 3. 数据入库
        XxxDO entity = XxxConvert.INSTANCE.convert(reqVO);
        xxxMapper.insert(entity);
        // 4. 关联表插入
        xxxRelMapper.insertRel(entity.getId(), ...);
        // 5. 操作日志记录
        operateLogApi.create(...);
        return entity.getId();
    }
}
```

### 5.4 适用范围

| 方法类型 | 必须加 @Transactional | 说明 |
|---------|----------------------|------|
| create（创建） | **是** | 新增数据涉及多表插入 |
| update（更新） | **是** | 更新数据涉及多表修改 |
| delete（删除） | **是** | 删除数据涉及关联检查 |
| query / read（查询） | **否** | 只读操作不需要事务 |

## 6. 参数校验规范

### 6.1 Service 层校验（项目主力校验层）

**约定: 在 Service 接口方法的参数上标注 `@Valid`，在 Service 实现类上标注 `@Validated`。**

```java
// ─── Service 接口 ───
public interface XxxService {
    Long create(@Valid XxxSaveReqVO createReqVO);   // 参数标注 @Valid
    void update(@Valid XxxSaveReqVO updateReqVO);
}

// ─── Service 实现 ───
@Service
@Validated                                            // 类级别触发方法参数校验
public class XxxServiceImpl implements XxxService {
    // ...
}
```

**校验流程**: 当 `XxxServiceImpl.create()` 被调用时，Spring AOP 拦截器检测到类上的 `@Validated` 和参数上的 `@Valid`，在校验失败时抛出 `ConstraintViolationException`，由 `GlobalExceptionHandler` 统一处理为 400 响应。

### 6.2 Controller 层校验

Controller 层也可以使用 `@Validated`（类级别）+ `@Valid`（参数级别），但 xy-cloud 的主要校验推力在 Service 层。Controller 主要负责请求参数的绑定和基础格式校验。

### 6.3 分组校验

当同一个 VO 在不同场景下需要不同的校验规则时，使用分组校验:

```java
public class AuthLoginReqVO {
    @NotEmpty(message = "验证码不能为空", groups = CodeEnableGroup.class)
    private String captchaVerification;

    // 定义分组接口
    public interface CodeEnableGroup {}
}

// 使用时在 @Validated 中指定分组
public interface AuthService {
    void login(@Validated(AuthLoginReqVO.CodeEnableGroup.class) AuthLoginReqVO reqVO);
}
```

**说明**: 分组接口仅作为标记，无需实现任何方法。JSR-380（Bean Validation 2.0）通过 `groups` 属性控制哪些约束在特定场景下生效。

### 6.4 自定义校验注解

所有自定义注解位于 `cn.shutan.platform.framework.common.validation` 包下:

| 注解 | 验证器类 | 用途 | 示例 |
|------|---------|------|------|
| `@InEnum` | `InEnumValidator` / `InEnumCollectionValidator` | 校验整数值是否在指定枚举范围内 | `@InEnum(value = GenderEnum.class)` |
| `@Mobile` | `MobileValidator` | 校验手机号格式（中国大陆） | `@Mobile` |
| `@Telephone` | `TelephoneValidator` | 校验座机号格式 | `@Telephone` |

**`@InEnum` 的特别说明**: `InEnumValidator` 校验单个枚举值，`InEnumCollectionValidator` 校验枚举值集合（如 `List<Integer>`）。两者通过 `@InEnum` 注解的 `isCollection()` 属性区分。

### 6.5 编程式校验

**类路径**: `cn.shutan.platform.framework.common.util.validation.ValidationUtils`

```java
// 手动触发校验，可指定分组
ValidationUtils.validate(object, groups...);

// 示例
ValidationUtils.validate(createReqVO);
ValidationUtils.validate(updateReqVO, UpdateGroup.class);
```

底层通过 `Validation.buildDefaultValidatorFactory().getValidator()` 获取 Validator 实例,调用失败时抛出 `ConstraintViolationException`。

**适用场景**: 当需要在 Service 方法内部按条件校验（如根据不同的业务分支校验不同的字段）时使用，替代在参数上声明式校验。

## 7. 统一响应模型 CommonResult

**类路径**: `cn.shutan.platform.framework.common.pojo.CommonResult`

```java
public class CommonResult<T> implements Serializable {
    private Integer code;    // 0=成功，非0=错误码
    private String msg;      // 错误消息（用户可读）
    private T data;          // 响应数据
}
```

### 7.1 核心 API

| 方法 | 作用 |
|------|------|
| `CommonResult.success(data)` | 创建成功响应，`code=0`, `msg=""` |
| `CommonResult.error(code, msg)` | 创建错误响应（断言确保 code != 0） |
| `CommonResult.error(errorCode)` | 通过 `ErrorCode` 对象创建错误响应 |
| `CommonResult.error(ServiceException)` | 通过 `ServiceException` 创建错误响应 |
| `result.isSuccess()` | 判断是否成功（`@JsonIgnore` 避免序列化） |
| `result.isError()` | 判断是否失败 |
| `result.checkError()` | 如果失败则抛出 `ServiceException`（跨服务调用检查） |
| `result.getCheckedData()` | 检查并获取数据 |

### 7.2 跨服务调用时的异常转换

```java
CommonResult<UserRespDTO> result = userApi.getUser(id);

// 方式一: checkError() —— 自动将 code != 0 转换为 ServiceException
result.checkError();

// 方式二: getCheckedData() —— 检查后直接获取 data（更简洁）
UserRespDTO user = result.getCheckedData();
```

**设计意图**: 在微服务架构中，服务间通过 Feign/Dubbo 调用时，返回的始终是 `CommonResult`。调用方需要手动检查 `code` 是否为 0。`checkError()` 将这一检查标准化为抛出 `ServiceException`，使得上层 `GlobalExceptionHandler` 能够统一处理。

### 7.3 类型转换

```java
// 将 A 方法的 CommonResult<A> 转换为 B 方法的 CommonResult<B>
// 常用于 Feign 调用链中的类型适配
CommonResult<T> result = CommonResult.error(anotherResult);
```

## 8. 异常流程架构

```
HTTP 请求
    │
    ▼
┌─────────────────────────────────────────────────┐
│  Filter 链（含 Spring Security）                  │
│  异常 → allExceptionHandler() 统一处理            │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│  Controller (@Validated + @Valid)                │
│  参数绑定失败:                                    │
│    @RequestParam 缺失 → MissingServletRequest... │
│    类型转换失败    → MethodArgumentTypeMismatch  │
│    @RequestBody   → HttpMessageNotReadable       │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│  Service 接口层 (@Valid 参数 + @Validated 类)     │
│  校验失败 → ConstraintViolationException         │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────┐
│  Service 业务逻辑                                │
│    ├─ 业务校验失败 → ServiceExceptionUtil        │
│    │   └── ServiceException                     │
│    │       └── GlobalExceptionHandler           │
│    │           ├─ 白名单消息: 不打印日志          │
│    │           ├─ 非白名单: warn + 首行栈        │
│    │           └─ CommonResult(code, msg)       │
│    │                                            │
│    ├─ 权限不足 → AccessDeniedException          │
│    │   └── GlobalExceptionHandler               │
│    │       └─ CommonResult(403, "没有该操作权限") │
│    │                                            │
│    └─ 未捕获异常 → Exception                    │
│        └── GlobalExceptionHandler               │
│            ├─ 表不存在处理 → 501 友好提示        │
│            ├─ error 日志记录                     │
│            ├─ 异步插入 ApiErrorLog               │
│            └─ CommonResult(500, "系统异常")     │
└─────────────────────────────────────────────────┘
```

### 流程关键节点说明

1. **Filter 层异常**: Filter（如 Spring Security Filter）执行中抛出的异常不经过 `@RestControllerAdvice`，而是由 `allExceptionHandler()` 静态方法统一处理，保证与 Controller 层异常处理逻辑一致。

2. **Controller 参数绑定**: Spring MVC 参数绑定阶段的异常（参数缺失、类型不匹配、`@RequestBody` 反序列化失败）在进入 Service 层之前即被捕获，返回 400 错误。

3. **Service 校验层**: 接口参数校验是项目的主力校验层。`@Validated` + `@Valid` 的组合确保进入业务逻辑前所有输入参数合法。

4. **Service 业务逻辑**: `ServiceExceptionUtil.exception()` 是抛出业务异常的标准化入口。禁止直接 `new ServiceException()`。

5. **兜底处理**: 未预期的 `Exception` 会被记录完整的 API 错误日志（包括请求参数、用户信息、异常堆栈等），写入数据库供后续排查。
