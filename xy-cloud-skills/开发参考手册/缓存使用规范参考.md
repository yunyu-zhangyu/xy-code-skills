# xy-cloud 缓存使用规范参考

> 技术栈: Guava Cache (LoadingCache) + Spring Cache (Redis) + 程序化 Redis DAO (StringRedisTemplate / Redisson)
> 涉及 Starter: platform-spring-boot-starter-redis, platform-common, platform-spring-boot-starter-protection

---

## 1. 缓存技术全景

xy-cloud 使用三层缓存架构，按数据一致性要求和访问频率选择不同模式：

| 模式 | 存储位置 | 一致性 | 适用场景 | 代表类/注解 |
|------|----------|--------|----------|-------------|
| Guava Local Cache | JVM 堆内 | 进程级，最终一致 | 高频读、低频写、可容忍节点间不一致 | CacheUtils |
| Spring Cache (Redis) | Redis | 分布式一致 | 标准 CRUD 读多写少、需跨节点一致 | @Cacheable / @CacheEvict |
| 程序化 Redis DAO | Redis | 强一致（原子操作） | 动态 TTL、复杂键模式、需要原子操作 | IdempotentRedisDAO 等 |

### 选型原则

- 数据变更不频繁、允许各节点短暂不一致 → **Guava Local Cache**（如字典数据、租户信息）
- 需要跨节点实时一致、标准 Key-Value 读写 → **Spring @Cacheable**（如角色、菜单、模板）
- 需要动态 TTL、原子 SET NX、令牌桶等特殊操作 → **程序化 Redis DAO**（如幂等、限流、令牌）

## 2. Guava Local Cache (CacheUtils)

### 2.1 CacheUtils API

**文件**: `platform-common/util/cache/CacheUtils.java`

```java
// 异步刷新缓存：过期时一个线程阻塞加载，其他线程返回旧值（stale-while-reload）
public static <K, V> LoadingCache<K, V> buildAsyncReloadingCache(
        Duration duration, CacheLoader<K, V> loader)

// 同步刷新缓存
public static <K, V> LoadingCache<K, V> buildCache(
        Duration duration, CacheLoader<K, V> loader)
```

- 底层基于 Guava `LoadingCache` + `refreshAfterWrite`
- 异步方式使用 `Executors.newCachedThreadPool()` 线程池

### 2.2 使用场景

| 使用位置 | 缓存内容 | TTL | 方式 |
|----------|----------|-----|------|
| SecurityFrameworkServiceImpl | 权限和角色检查结果 | 1 min | buildAsyncReloadingCache |
| TenantFrameworkServiceImpl | 租户 ID 列表、租户验证结果 | 1 min | buildAsyncReloadingCache |
| DictFrameworkUtils | 字典数据 label/value 查找 | 1 min | buildAsyncReloadingCache |
| SocialClientServiceImpl | WxMpService / WxMaService 实例 | 10 s | buildCache |

### 2.3 完整示例

```java
// SecurityFrameworkServiceImpl
private final LoadingCache<Long, Set<String>> permissionCache =
        CacheUtils.buildAsyncReloadingCache(Duration.ofMinutes(1),
                new CacheLoader<Long, Set<String>>() {
                    @Override
                    public Set<String> load(Long userId) {
                        return permissionService.getUserPermissionListFromCache(userId);
                    }
                });

public Set<String> getPermissionCache(Long userId) {
    return permissionCache.getUnchecked(userId);
}
```

> **说明**: Guava Local Cache 适用于"系统范围"或"全局"数据，不要求跨节点强一致性。使用 `refreshAfterWrite` 而非 `expireAfterWrite`，保证缓存过期时仍有旧值可用。

## 3. Spring Cache via Redis

### 3.1 自动配置

#### PlatformRedisAutoConfiguration

**文件**: `platform-spring-boot-starter-redis/config/PlatformRedisAutoConfiguration.java`

- 创建 `RedisTemplate<String, Object>` bean
- Key 序列化: `StringRedisSerializer`
- Value 序列化: `Jackson2JsonRedisSerializer`（带 `JavaTimeModule` 修复 LocalDateTime）
- 在 `RedissonAutoConfiguration` 之前运行

#### PlatformCacheAutoConfiguration

**文件**: `platform-spring-boot-starter-redis/config/PlatformCacheAutoConfiguration.java`

- 启用 `@EnableCaching`
- 创建 `RedisCacheConfiguration`，将 key 分隔符从 `::`（Spring 默认）改为 `:`（Redis Desktop Manager 兼容）
- JSON 值序列化
- 全局 TTL 从 `spring.cache.redis.time-to-live` 读取
- 创建 `TimeoutRedisCacheManager`

### 3.2 TimeoutRedisCacheManager（缓存级 TTL）

**文件**: `platform-spring-boot-starter-redis/core/TimeoutRedisCacheManager.java`

扩展 `RedisCacheManager`，支持在 `cacheNames` 中嵌入 TTL：

```
格式: cacheName#ttl
示例: "role#10m", "session#1h", "token#30s", "data#1d"
```

| 后缀 | 单位 |
|------|------|
| d | 天 |
| h | 小时 |
| m | 分钟 |
| s | 秒（默认） |

### 3.3 TenantRedisCacheManager（租户隔离）

**文件**: `platform-spring-boot-starter-biz-tenant/core/redis/TenantRedisCacheManager.java`

扩展 `TimeoutRedisCacheManager`，在缓存名称后追加 `:{tenantId}`：

```
原始 key: role:123
租户隔离后: role:123:{tenantId}
```

可通过 `ignoreCaches` 配置跳过租户后缀的缓存列表。

### 3.4 RedisKeyConstants（缓存名称管理）

**文件**: `platform-module-system-biz/dal/redis/RedisKeyConstants.java`

使用 **interface 常量模式** 统一管理所有缓存名称：

```java
public interface RedisKeyConstants {

    // ========== 部门 ==========
    String DEPT_CHILDREN_ID_LIST = "dept_children_ids";

    // ========== 角色 ==========
    String ROLE = "role";
    String USER_ROLE_ID_LIST = "user_role_ids";
    String MENU_ROLE_ID_LIST = "menu_role_ids";
    String PERMISSION_MENU_ID_LIST = "permission_menu_ids";

    // ========== OAuth2 ==========
    String OAUTH_CLIENT = "oauth_client";
    String OAUTH2_ACCESS_TOKEN = "oauth2_access_token";

    // ========== 消息模板 ==========
    String NOTIFY_TEMPLATE = "notify_template";
    String MAIL_ACCOUNT = "mail_account";
    String MAIL_TEMPLATE = "mail_template";
    String SMS_TEMPLATE = "sms_template";

    // ========== 微信 ==========
    String WXA_SUBSCRIBE_TEMPLATE = "wxa_subscribe_template";
}
```

> **规则**: 常量名 = 业务含义的蛇形/小驼峰，值 = 纯英文小写 + 下划线，不加冒号（由框架自动拼接）。

### 3.5 @Cacheable 读取模式

```java
// 标准读取：结果为 null 时不缓存（防缓存穿透）
@Override
@Cacheable(value = RedisKeyConstants.ROLE, key = "#id",
        unless = "#result == null")
public RoleDO getRoleFromCache(Long id) {
    return roleMapper.selectById(id);
}

// 禁用数据权限 + 缓存
@Override
@DataPermission(enable = false)
@Cacheable(cacheNames = RedisKeyConstants.DEPT_CHILDREN_ID_LIST, key = "#id")
public Set<Long> getChildDeptIdListFromCache(Long id) {
    return ...;
}
```

### 3.6 @CacheEvict 失效模式

项目采用 **写失效** 策略（而不是缓存更新），修改数据时直接清除缓存：

```java
// 单条失效：精确指定 key
@CacheEvict(value = RedisKeyConstants.ROLE, key = "#updateReqVO.id")
public void updateRole(RoleSaveReqVO updateReqVO) { ... }

// 整表失效：清除该 cacheName 下所有条目
@CacheEvict(cacheNames = RedisKeyConstants.OAUTH_CLIENT, allEntries = true)
public void updateOAuth2Client(OAuth2ClientSaveReqVO reqVO) { ... }

// 条件失效：满足条件才失效
@CacheEvict(value = RedisKeyConstants.PERMISSION_MENU_ID_LIST,
        key = "#createReqVO.permission",
        condition = "#createReqVO.permission != null")
public Long createMenu(MenuSaveVO createReqVO) { ... }
```

### 3.7 @Caching 批量失效

```java
@Override
@Caching(evict = {
    @CacheEvict(value = RedisKeyConstants.MENU_ROLE_ID_LIST, allEntries = true),
    @CacheEvict(value = RedisKeyConstants.PERMISSION_MENU_ID_LIST, allEntries = true)
})
public void assignRoleMenu(Long roleId, Set<Long> menuIds) { ... }
```

### 3.8 完整 CRUD 缓存示例

以角色管理为例，展示完整的缓存注解使用模式：

```java
@Service
public class RoleServiceImpl implements RoleService {

    // CREATE — 新增不需要读缓存，无需注解
    @Override
    public Long create(RoleSaveReqVO reqVO) { ... }

    // UPDATE — 失效单条缓存
    @Override
    @CacheEvict(value = RedisKeyConstants.ROLE, key = "#updateReqVO.id")
    public void update(RoleSaveReqVO updateReqVO) { ... }

    // DELETE — 失效单条缓存
    @Override
    @CacheEvict(value = RedisKeyConstants.ROLE, key = "#id")
    public void delete(Long id) { ... }

    // GET — 读取优先查缓存，miss 回源 DB
    @Override
    @Cacheable(value = RedisKeyConstants.ROLE, key = "#id",
            unless = "#result == null")
    public RoleDO getFromCache(Long id) {
        return roleMapper.selectById(id);
    }
}
```

## 4. 程序化 Redis DAO

当 Spring Cache 注解无法满足需求时（动态 TTL、原子操作、复杂键模式），使用专门的 Redis DAO 类。

### 4.1 DAO 类结构模板

```java
@Component
public class XxxRedisDAO {

    private final StringRedisTemplate stringRedisTemplate;

    // KEY 前缀常量
    private static final String KEY_PREFIX = "xxx:";

    public XxxRedisDAO(StringRedisTemplate stringRedisTemplate) {
        this.stringRedisTemplate = stringRedisTemplate;
    }

    // 构建完整 key
    private static String formatKey(String suffix) {
        return KEY_PREFIX + suffix;
    }

    // 业务读写方法
    public void set(String id, String value, long timeout, TimeUnit unit) {
        stringRedisTemplate.opsForValue().set(formatKey(id), value, timeout, unit);
    }

    public String get(String id) {
        return stringRedisTemplate.opsForValue().get(formatKey(id));
    }

    public void delete(String id) {
        stringRedisTemplate.delete(formatKey(id));
    }
}
```

### 4.2 OAuth2AccessTokenRedisDAO — 动态 TTL

**文件**: `platform-module-system-biz/dal/redis/oauth2/OAuth2AccessTokenRedisDAO.java`

```java
@Component
public class OAuth2AccessTokenRedisDAO {

    private final StringRedisTemplate stringRedisTemplate;

    private static final String KEY_PREFIX = "oauth2_access_token:";

    public void set(OAuth2AccessTokenDO accessTokenDO) {
        String redisKey = KEY_PREFIX + accessTokenDO.getAccessToken();
        // TTL 使用令牌本身的过期时间（动态计算）
        long time = LocalDateTimeUtil.between(
                LocalDateTime.now(),
                accessTokenDO.getExpiresTime(),
                ChronoUnit.SECONDS);
        if (time > 0) {
            stringRedisTemplate.opsForValue().set(
                    redisKey,
                    JsonUtils.toJsonString(accessTokenDO),
                    time, TimeUnit.SECONDS);
        }
    }

    public OAuth2AccessTokenDO get(String accessToken) {
        String value = stringRedisTemplate.opsForValue().get(KEY_PREFIX + accessToken);
        return JsonUtils.parseObject(value, OAuth2AccessTokenDO.class);
    }

    public void delete(String accessToken) {
        stringRedisTemplate.delete(KEY_PREFIX + accessToken);
    }
}
```

### 4.3 IdempotentRedisDAO — 幂等控制

**文件**: `platform-spring-boot-starter-protection/idempotent/core/redis/IdempotentRedisDAO.java`

```java
@Component
public class IdempotentRedisDAO {

    private static final String KEY_PREFIX = "idempotent:";

    // SET NX EX — 原子操作，不存在则设置
    public Boolean setIfAbsent(String key, long timeout, TimeUnit timeUnit) {
        String redisKey = KEY_PREFIX + key;
        return redisTemplate.opsForValue().setIfAbsent(redisKey, "", timeout, timeUnit);
    }
}
```

由 `@Idempotent` 注解使用，默认 timeout = 1 秒。

## 5. 保护注解 (@Idempotent / @RateLimiter / @ApiSignature)

这三个注解由 `platform-spring-boot-starter-protection` 提供，底层基于 Redis。

### 5.1 @Idempotent — 幂等控制

```java
@Idempotent(timeout = 1, timeUnit = TimeUnit.SECONDS, keyResolver = IdempotentKeyResolverEnum.USER)
```

| 属性 | 默认值 | 说明 |
|------|--------|------|
| timeout | 1 | 超时时间 |
| timeUnit | TimeUnit.SECONDS | 时间单位 |
| keyResolver | IdempotentKeyResolverEnum.DEFAULT | Key 解析器：DEFAULT / USER / EXPRESSION |
| message | "重复请求，请稍后重试" | 提示消息 |

### 5.2 @RateLimiter — 接口限流

```java
@RateLimiter(count = 100, time = 1, timeUnit = TimeUnit.SECONDS, keyResolver = RateLimiterKeyResolverEnum.CLIENT_IP)
```

| 属性 | 默认值 | 说明 |
|------|--------|------|
| count | 100 | 最大请求数 |
| time | 1 | 时间窗口 |
| timeUnit | TimeUnit.SECONDS | 时间单位 |
| keyResolver | RateLimiterKeyResolverEnum.DEFAULT | Key 解析器：DEFAULT / USER / CLIENT_IP / SERVER_NODE / EXPRESSION |
| message | "请求过于频繁" | 提示消息 |

### 5.3 @ApiSignature — API 签名校验

```java
@ApiSignature
```

用于接口签名校验，包含 nonce 防重放、timestamp 校验、sign 校验。

## 6. 缓存选型决策

### 场景对照表

| 业务场景 | 推荐模式 | 原因 |
|----------|----------|------|
| 字典数据查询（高频读、低频写） | Guava Local Cache | 容忍节点间不一致，延迟最低 |
| 权限/角色检查 | Guava Local Cache | 每次请求都查，需要极低延迟 |
| 租户信息校验 | Guava Local Cache | 变化极少，最终一致即可 |
| 标准实体缓存（角色/菜单/OAuth 客户端） | Spring @Cacheable | 需要跨节点一致，标准 KV 读写 |
| 需要动态 TTL 的数据（OAuth2 令牌） | 程序化 Redis DAO | TTL 跟随业务数据过期时间 |
| 分布式锁/幂等/SET NX EX | 程序化 Redis DAO | 需要原子操作 |
| 分布式限流（令牌桶） | 程序化 Redis DAO（Redisson） | Redisson RRateLimiter 原生支持 |

### 性能考量

- Guava Local Cache: 微秒级读延迟，不消耗网络，适用于高频读
- Spring Cache (Redis): 毫秒级读延迟（网络 RTT + Redis 操作），需序列化/反序列化
- 程序化 Redis DAO: 同 Redis 操作延迟，适用于特殊原子操作

### 7.2 Redis Key 命名规范

| 分类 | Key 模式 | 示例 |
|------|----------|------|
| 业务缓存 | `{cacheName}:{entityId}` | `role:123` |
| 多租户缓存 | `{cacheName}:{entityId}:{tenantId}` | `role:123:1` |
| 幂等控制 | `idempotent:{key}` | `idempotent:user:123` |
| 限流 | `rate_limiter:{key}` | `rate_limiter:ip:192.168.x.x` |
| API 签名 nonce | `api_signature_nonce:{appId}:{nonce}` | `api_signature_nonce:app1:abc123` |
| 分布式锁 | `lock4j:{key}` | `lock4j:order:456` |
