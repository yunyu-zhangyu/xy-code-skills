## 1. 权限控制体系

### 1.1 权限注解 @PreAuthorize

项目使用 `@PreAuthorize` 配合自定义 Spring EL 表达式 `@ss.hasPermission()` 进行接口权限控制。

**权限字符串格式**: `'{module}:{entity}:{action}'`

| 段 | 说明 | 常见值 |
|---|---|---|
| `module` | 模块名 | system, infra, eos 等 |
| `entity` | 实体名 | user, dept, role, menu, config 等 |
| `action` | 操作 | create, update, delete, query, export |

**典型使用示例**:

```java
@PreAuthorize("@ss.hasPermission('system:user:create')")
@PreAuthorize("@ss.hasPermission('system:user:query')")
@PreAuthorize("@ss.hasPermission('system:dept:update')")
@PreAuthorize("@ss.hasPermission('system:dept:delete')")
@PreAuthorize("@ss.hasPermission('system:permission:assign-role-menu')")
@PreAuthorize("@ss.hasPermission('infra:config:create')")
@PreAuthorize("@ss.hasPermission('infra:job:query')")
@PreAuthorize("@ss.hasPermission('eos:weather:create')")
@PreAuthorize("@ss.hasPermission('eos:alarm-rule:export')")
```

> **说明**: `@ss` 是 `SecurityFrameworkServiceImpl` 在 Spring EL 中的 Bean 名称。当注解标注在 Controller 方法上时，Spring Security 会在方法调用前执行 EL 表达式，调用 `@ss.hasPermission()` 进行权限校验，校验不通过则抛出 `AccessDeniedException`，最终由 `AccessDeniedHandlerImpl` 返回 403 响应。

### 1.2 免登录注解 @PermitAll

对于登录、注册、验证码、回调等公开接口，使用 `@PermitAll` 注解标注，跳过认证和权限校验。

```java
// 用于登录、注册、验证码、回调等公开接口
import javax.annotation.security.PermitAll;

@PermitAll
@PostMapping("/login")
public CommonResult<AuthLoginRespVO> login(@Valid @RequestBody AuthLoginReqVO reqVO) {
    // 登录逻辑，无需 Token
}
```

> **说明**: `@PermitAll` 是 JSR-250 规范的注解，Spring Security 默认支持。与 `@PreAuthorize("permitAll()")` 效果相同，但语义更清晰。项目中还通过 `SecurityProperties.permit-all-urls` 配置额外的免登录 URL 列表。

### 1.3 权限校验服务

权限校验的核心接口和实现类如下：

- **接口**: `cn.shutan.platform.framework.security.core.service.SecurityFrameworkService`
  - 定义了以下核心方法：`hasPermission`, `hasAnyPermissions`, `hasRole`, `hasAnyRoles`, `hasScope`, `hasAnyScopes`
- **实现类**: `cn.shutan.platform.framework.security.core.service.SecurityFrameworkServiceImpl`
  - 在 Spring EL 中以 Bean 名称 `@ss` 被引用

**调用链路**:

```
@PreAuthorize("@ss.hasPermission('system:user:create')")
    → SecurityFrameworkServiceImpl.hasPermission()
        → PermissionApi (Feign 客户端)
            → 远程权限校验服务
        → Guava Cache (结果缓存 1 分钟)
```

> **说明**: 权限校验通过 Feign 调用远程服务完成，而非本地校验。这意味着权限数据集中管理，各服务统一通过 `PermissionApi` 获取校验结果。校验结果使用 Guava Cache 缓存 1 分钟，避免频繁的远程调用，在实时性和性能之间取得平衡。

## 2. 用户上下文

### 2.1 LoginUser 模型

`cn.shutan.platform.framework.security.core.LoginUser`

这是项目中代表当前登录用户的核心模型类，继承自 Spring Security 的 `UserDetails`，包含了认证和授权所需的全部信息。

**核心字段**:

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long | 用户 ID |
| userType | Integer | 用户类型（如管理员、普通用户等） |
| tenantId | Long | 租户 ID |
| scopes | List\<String\> | 权限范围列表 |
| expiresTime | LocalDateTime | 过期时间 |
| info | Map\<String, String\> | 扩展信息（nickname 昵称、deptId 部门 ID 等） |
| context | Map\<String, Object\> | 临时缓存，请求生命周期内共享数据 |

> **说明**: `info` 字段用于存放用户的基本属性（如昵称、部门 ID），而 `context` 用于存放请求生命周期内的临时数据，两者职责不同。`LoginUser` 会被序列化存储在 SecurityContext 中，并在 Feign 调用时通过 `login-user` Header 透传给下游服务。

### 2.2 SecurityFrameworkUtils（服务端工具类）

`cn.shutan.platform.framework.security.core.util.SecurityFrameworkUtils`

提供一组静态方法，方便在服务端代码中获取当前用户信息：

```java
getLoginUser()                      // 获取当前 LoginUser 对象
getLoginUserId()                    // 获取当前用户 ID（快捷方法）
getLoginUserNickname()              // 获取当前用户昵称
getLoginUserDeptId()                // 获取当前用户部门 ID
setLoginUser(loginUser, request)    // 将 LoginUser 设置到 SecurityContext 中
obtainAuthorization(request, headerName, parameterName)  // 从请求中提取 Token
```

> **说明**: 这些工具方法封装了对 `SecurityContextHolder` 的操作，推荐在业务代码中使用而非直接操作 SecurityContext。`setLoginUser` 方法在认证通过后将用户信息写入 SecurityContext，供后续请求处理使用。

### 2.3 WebFrameworkUtils（通用工具类）

`cn.shutan.platform.framework.web.core.util.WebFrameworkUtils`

虽然不专属于安全模块，但提供了获取用户信息的便捷方法，在 Web 层通用：

```java
getLoginUserId(request)     // 从请求中获取登录用户 ID
getLoginUserType(request)   // 从请求中获取用户类型
getTenantId(request)        // 从请求中获取租户 ID
```

> **说明**: 与 `SecurityFrameworkUtils` 的区别在于，`WebFrameworkUtils` 直接从 `HttpServletRequest` 的 Attribute 中取值，适用于 Filter 或 Interceptor 等还未建立完整 SecurityContext 的场景。

### 2.4 Feign 透传机制

`cn.shutan.platform.framework.security.core.rpc.LoginUserRequestInterceptor`

**作用**: Feign 请求拦截器，在微服务间 RPC 调用时自动传递用户上下文。

**工作流程**:

```
服务 A (当前请求)
    → LoginUserRequestInterceptor 拦截 Feign 请求
        → 从 SecurityContextHolder 获取 LoginUser
        → 序列化为 JSON 字符串
        → 写入请求 Header: login-user
    → 服务 B (接收请求)
        → LoginUserRequestInterceptor 读取 login-user Header
        → 反序列化为 LoginUser 对象
        → 设置到当前线程的 SecurityContextHolder
```

> **说明**: 该机制实现了用户上下文在微服务间的透明传递，下游服务无需再次校验 Token 即可获取当前用户信息。但需要注意安全风险——仅应在内部可信网络中使用此机制，Gateway 层应确保外部请求不能伪造 `login-user` Header。

## 3. 数据权限 @DataPermission

### 3.1 注解定义

`cn.shutan.platform.framework.datapermission.core.annotation.DataPermission`

用于控制 MyBatis Plus 查询时是否应用数据权限过滤，标注在 Service 方法或 Mapper 方法上。

**注解属性**:

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| enable | boolean | true | 是否启用数据权限过滤 |
| includeRules | Class\<\?\>[] | \{\} | 指定仅生效的规则类（白名单模式） |
| excludeRules | Class\<\?\>[] | \{\} | 排除的规则类（黑名单模式） |

### 3.2 典型使用场景

```java
// 场景一：关闭数据权限，避免递归查询
// 在查询部门数据权限时，如果仍开启数据权限会导致递归
@DataPermission(enable = false)
public DeptDataPermissionRespDTO getDeptDataPermission(Long userId) {
    // 查询用户部门数据权限配置
}

// 场景二：缓存方法避免数据权限干扰
// 缓存操作应基于完整数据，不应受数据权限限制
@DataPermission(enable = false)
public String getParentDeptPath(Long id) {
    // 获取部门路径
}

// 场景三：编程式忽略数据权限
DataPermissionUtils.executeIgnore(() -> {
    // 此代码块内的数据查询忽略数据权限
    List<UserDO> allUsers = userMapper.selectList(...);
});
```

> **说明**: 数据权限的核心场景是为不同角色的用户提供不同范围的数据可见性（如部门经理看全部门数据，普通员工只看自己）。但在某些场景下（如获取数据权限配置本身、缓存数据加载等），需要临时关闭数据权限，否则会导致递归或结果不完整。

### 3.3 核心机制

数据权限的实现依赖于以下几个组件的协同工作：

| 组件 | 类名 | 职责 |
|------|------|------|
| AOP 拦截器 | `DataPermissionAnnotationInterceptor` | 拦截标注了 `@DataPermission` 的方法，将注解信息推入/弹出栈 |
| 上下文持有者 | `DataPermissionContextHolder` | 基于 ThreadLocal 的栈结构，维护当前线程的数据权限上下文 |
| MyBatis Plus 处理器 | `DataPermissionRuleHandler` | 实现 `MultiDataPermissionHandler`，在 SQL 执行前拼接数据权限条件 |
| 编程式工具 | `DataPermissionUtils` | 提供 `executeIgnore()` 等静态方法，便捷地临时关闭数据权限 |

**执行流程**:

```
方法调用
    → DataPermissionAnnotationInterceptor AOP 拦截
        → 解析 @DataPermission 注解
        → DataPermissionContextHolder 入栈
    → MyBatis Plus 执行 SQL
        → DataPermissionRuleHandler 被调用
            → 从 ContextHolder 获取当前注解配置
            → 根据规则拼接 WHERE 条件
    → 方法返回
        → ContextHolder 出栈，恢复之前的状态
```

### 3.4 部门数据权限规则

`DeptDataPermissionRule` 是最常用的数据权限规则实现，根据当前用户的部门数据权限设置，自动拼接 SQL WHERE 条件。

**规则行为**:

| 用户权限设置 | SQL 拼接效果 | 说明 |
|-------------|-------------|------|
| 全部可见 | 不拼接额外条件 | 管理员或拥有全部数据权限的用户 |
| 部分部门可见 | `dept_id IN (可见部门 ID 列表)` | 仅查看指定部门的数据 |
| 仅自己可见 | `user_id = 当前用户 ID` | 仅查看自己的数据 |

**自定义扩展**: 通过 `DeptDataPermissionRuleCustomizer` 接口，可以为不同表配置不同的部门字段名和用户字段名：

```java
@Bean
public DeptDataPermissionRuleCustomizer customizer() {
    return rule -> {
        rule.addDeptColumn("sys_user", "dept_id");      // sys_user 表使用 dept_id 字段
        rule.addUserColumn("sys_order", "creator_id");  // sys_order 表使用 creator_id 字段
    };
}
```

> **说明**: 不同业务的表结构可能不同，部门字段可能叫 `dept_id`、`department_id`，用户字段可能叫 `user_id`、`creator_id`。通过 `DeptDataPermissionRuleCustomizer` 可以灵活适配各种表结构。

## 4. 多租户

### 4.1 @TenantIgnore 注解

`cn.shutan.platform.framework.tenant.core.aop.TenantIgnore`

标注在方法上，忽略租户过滤，用于跨租户操作的场景。

```java
// 场景一：用户名校验需要跨租户
// 注册新用户时，需要检查用户名在所有租户中是否唯一
@TenantIgnore
private AdminUserDO validateUserForCreateOrUpdate(String username) {
    // 跨租户查询用户名是否已被使用
}

// 场景二：租户管理自身的分页查询
// 查询租户列表时不能受租户过滤限制（否则只能查到自己）
@TenantIgnore
public PageResult<TenantDO> getTenantPage(TenantPageReqVO pageReqVO) {
    // 查询所有租户
}
```

> **说明**: 多租户模式下，默认所有 SQL 查询都会自动拼接 `tenant_id = ?` 条件。但某些操作（如用户名唯一性校验、租户列表查询等）需要在全局范围内执行，此时需要使用 `@TenantIgnore` 或 `TenantUtils.executeIgnore()` 临时关闭租户过滤。

### 4.2 TenantUtils 工具类

`cn.shutan.platform.framework.tenant.core.util.TenantUtils`

提供编程式的租户上下文切换方法：

```java
// 指定租户 ID 执行（适用于定时任务、消息处理等需要明确租户的场景）
TenantUtils.execute(tenantId, () -> {
    // 在此租户上下文中执行操作
    orderService.listOrders();
});

// 指定租户 ID 执行（带返回值）
OrderDO order = TenantUtils.execute(tenantId, () -> {
    return orderService.getOrder(orderId);
});

// 忽略租户过滤执行
TenantUtils.executeIgnore(() -> {
    // 跨租户操作
    List<UserDO> allUsers = userMapper.selectList(...);
});
```

### 4.3 租户过滤机制

多租户过滤通过 MyBatis Plus 拦截器自动实现，核心组件如下：

| 组件 | 类名 | 职责 |
|------|------|------|
| 数据库拦截器 | `TenantDatabaseInterceptor` | 实现 `TenantLineHandler`，自动为 SQL 拼接 `tenant_id = ?` |
| 上下文持有者 | `TenantContextHolder` | 基于 TransmittableThreadLocal，解决线程池中租户上下文传递问题 |
| Web 过滤器 | `TenantContextWebFilter` | 从请求 Header `tenant-id` 提取租户 ID，设置到 ContextHolder |
| 安全过滤器 | `TenantSecurityWebFilter` | 校验租户合法性，防止越权访问已禁用或过期的租户 |

**执行流程**:

```
HTTP 请求
    → TenantContextWebFilter
        → 从 tenant-id Header 提取租户 ID
        → 设置到 TenantContextHolder
    → TenantSecurityWebFilter
        → 校验租户是否合法（未禁用、未过期）
        → 不合法则拒绝请求
    → 业务逻辑执行 SQL
        → TenantDatabaseInterceptor 拦截
            → 从 ContextHolder 获取当前租户 ID
            → 自动拼接 tenant_id = ?
    → 请求结束
        → 清理 ContextHolder
```

> **说明**: 使用 `TransmittableThreadLocal`（阿里巴巴开源）而非普通 `ThreadLocal`，是因为在异步场景（如线程池）中，父线程的租户上下文需要能够传递给子线程。`TransmittableThreadLocal` 解决了这个线程池上下文传递问题。

### 4.4 租户忽略配置

某些表在所有租户间共享（如系统配置表、字典表等），不应进行租户过滤。通过以下配置指定：

```yaml
platform:
  tenant:
    ignore-tables:
      - sys_config
      - sys_dict_data
      - sys_dict_type
      - infra_job
```

> **说明**: 配置中指定的表名将在 `TenantDatabaseInterceptor` 中被跳过，不拼接 `tenant_id = ?` 条件。这是一种白名单黑名单之外的第三种方式——按表名忽略，适用于明确不需要租户隔离的基础数据表。

### 4.5 DO 继承要求（重要）

`TenantDatabaseInterceptor` 通过 MyBatis-Plus 的 `StatementHandler` 拦截 SQL 并自动拼接 `tenant_id = ?` 条件。**该拦截器依赖 DO 类的父类类型**来判断是否需要租户过滤：

- 继承 `cn.shutan.platform.framework.mybatis.core.dataobject.TenantBaseDO` → 触发租户过滤
- 继承 `cn.shutan.platform.framework.mybatis.core.dataobject.BaseDO` → **不触发**租户过滤

> ⚠️ **这是一处静默失败**：如果本应继承 `TenantBaseDO` 的 DO 错误地继承了 `BaseDO`，SQL 不会拼接租户过滤条件，**数据将在租户间泄露**，且不会产生任何异常或错误日志。

**各模块 DO 父类对照表**：

| 模块 | 推荐父类 | 原因 |
|------|---------|------|
| `platform-module-system` | `TenantBaseDO` | 多租户数据，需要租户隔离 |
| `platform-module-infra` | `TenantBaseDO` / `BaseDO` | 部分表跨租户共享，按实际情况选择 |
| `platform-module-eos` | `BaseDO` | eos 模块通常不涉及多租户 |

**排查方法**：如果怀疑数据泄露，检查对应 DO 的父类声明，确认是否继承自 `TenantBaseDO`。全局搜索 `extends BaseDO` 并在多租户表中交叉比对。

### 4.6 TenantContextHolder 完整生命周期

以下展示一次多租户请求从进入到返回期间，`TenantContextHolder` 中 `tenantId` 和 `ignore` 标志在每个阶段的值变化：

```
TenantContextHolder 状态变化（值 / ignore 标志）

请求进入
  │  tenantId=null, ignore=false
  ▼
① TenantContextWebFilter
  │  从 Header[tenant-id] 读取 → "t001"
  │  TenantContextHolder.setTenantId("t001")
  │  → tenantId="t001", ignore=false
  ▼
② TenantSecurityWebFilter（越权 + 合法性检查）
  │  检查: 登录用户的 tenantId == "t001" ?
  │  调用: TenantFrameworkServiceImpl.validTenant("t001")
  │  → tenantId="t001", ignore=false
  ▼
③ Controller → Service → Mapper
  │
  ├─ 正常路径：TenantDatabaseInterceptor
  │    getTenantId() 返回 "t001"
  │    ignoreTable() 返回 false
  │    → SQL 自动拼接: AND tenant_id = 't001'
  │
  ├─ @TenantIgnore 方法：TenantIgnoreAspect
  │    设置 ignore=true
  │    → SQL 不拼接 tenant_id
  │    方法结束后恢复 ignore=false
  │
  └─ TenantUtils.execute("t002", ...)
       保存旧值: tenantId="t001", ignore=false
       设置新值: tenantId="t002"
       执行完毕后恢复: tenantId="t001", ignore=false
  ▼
④ Feign 出站调用（TenantRequestInterceptor）
  │  从 ContextHolder 获取 "t001"
  │  写入 Header[tenant-id] = "t001"
  │  下游服务 → 从 ① 重新开始
  │  → tenantId="t001", ignore=false（当前线程不变）
  ▼
⑤ 异步线程（TransmittableThreadLocal 自动传递）
  │  父线程 → 子线程
  │  → 子线程 tenantId="t001", ignore=false
  │  子线程结束后 ContextHolder 清理
  ▼
⑥ 请求返回（TenantContextWebFilter finally）
  │  TenantContextHolder.clear()
  │  → tenantId=null, ignore=false
  ▼
响应返回客户端
```

**关键约束**：

- **DO 继承决策**：继承 `TenantBaseDO` 才触发租户过滤；继承 `BaseDO` 不触发。判断方法：检查表是否包含 `tenant_id` 列。
- **ignore 标志优先级**：`ignore=true` 时即使 `tenantId` 有值也不拼接 SQL。
- **异步传递**：`TransmittableThreadLocal` 自动向子线程传递上下文，但线程池需配置 `TtlExecutors`。

> ⚠️ **静默失败警告**：如果 DO 错误地继承了 `BaseDO`（而非 `TenantBaseDO`），SQL 不会拼接租户条件 → 数据在租户间泄露 → 无任何异常或日志。排查：全局搜索 `extends BaseDO`，交叉比对表是否含 `tenant_id` 列。

## 5. API 安全防护

### 5.1 @Idempotent 幂等注解

`cn.shutan.platform.framework.idempotent.core.annotation.Idempotent`

防止接口重复提交，基于 Redis 实现，适用于表单提交、支付等敏感操作。

**注解属性**:

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| timeout | int | 1 | 超时时间（在此时间内重复请求被视为重复提交） |
| timeUnit | TimeUnit | SECONDS | 时间单位 |
| message | String | "重复请求，请稍后重试" | 重复请求时的提示消息 |
| keyResolver | Class | DefaultIdempotentKeyResolver | Key 解析策略 |
| keyArg | String | "" | 自定义表达式参数（配合 ExpressionIdempotentKeyResolver 使用） |
| deleteKeyWhenException | boolean | true | 业务异常时是否删除 Key（true 则允许重试） |

**Key 解析策略**:

| 策略类 | 说明 | 适用场景 |
|--------|------|----------|
| `DefaultIdempotentKeyResolver` | 全局唯一 Key | 整个接口全局幂等 |
| `UserIdempotentKeyResolver` | 基于用户 ID | 同一用户对同一接口的重复提交 |
| `ExpressionIdempotentKeyResolver` | 自定义 SpEL 表达式 | 基于请求参数的精细控制 |

**使用示例**:

```java
@PostMapping("/create")
@Idempotent(timeout = 5, timeUnit = TimeUnit.SECONDS, message = "请勿重复提交")
public CommonResult<Long> create(@Valid @RequestBody XxxCreateReqVO reqVO) {
    // 5 秒内重复提交会被拦截
}
```

> **说明**: 幂等实现基于 Redis 的 SET NX 命令（不存在则写入）。首次请求写入一个 Key，超时时间内再次请求发现 Key 已存在则判定为重复提交。`deleteKeyWhenException` 属性控制业务异常（区别于重复提交）时是否释放 Key，设置为 true 表示业务失败后可重试。

### 5.2 @RateLimiter 限流注解

`cn.shutan.platform.framework.ratelimiter.core.annotation.RateLimiter`

防止接口被高频调用，同样基于 Redis 实现。

**注解属性**:

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| time | int | 1 | 时间窗口大小 |
| timeUnit | TimeUnit | SECONDS | 时间单位 |
| count | int | 100 | 时间窗口内允许的最大请求数 |
| message | String | "请求过于频繁" | 被限流时的提示消息 |
| keyResolver | Class | DefaultRateLimiterKeyResolver | Key 解析策略 |

**Key 解析策略**:

| 策略类 | 说明 | 适用场景 |
|--------|------|----------|
| `DefaultRateLimiterKeyResolver` | 全局 Key | 整个接口的总体限流 |
| `UserRateLimiterKeyResolver` | 基于用户 ID | 每个用户分别限流 |
| `ClientIpRateLimiterKeyResolver` | 基于客户端 IP | 按 IP 限流，防爬虫 |
| `ServerNodeRateLimiterKeyResolver` | 基于服务器节点 | 集群模式下按节点限流 |
| `ExpressionRateLimiterKeyResolver` | 自定义 SpEL 表达式 | 灵活的限流粒度控制 |

**使用示例**:

```java
@GetMapping("/page")
@RateLimiter(time = 1, timeUnit = TimeUnit.MINUTES, count = 1000,
             message = "请求频率过高，请稍后再试",
             keyResolver = UserRateLimiterKeyResolver.class)
public CommonResult<PageResult<XxxDO>> page(XxxPageReqVO reqVO) {
    // 每分钟每个用户最多 1000 次请求
}
```

> **说明**: 限流基于 Redis 的计数器（Sorted Set 或 String 计数器）实现，通过滑动窗口或固定窗口算法统计请求频率。两种注解的 Redis 操作分别封装在 `IdempotentRedisDAO` 和 `RateLimiterRedisDAO` 中。

## 6. OAuth2 Token 认证

### 6.1 Token 认证流程

项目使用 OAuth2 标准的 Token 认证机制，认证流程如下：

```
客户端请求（携带 Token）
    │
    ├─ Gateway 层:
    │   TokenAuthenticationFilter (GlobalFilter)
    │   ├─ 检查 login-user Header（网关已解析时的优化路径）
    │   ├─ 从 Authorization Header 或 token 参数提取 Token
    │   ├─ 调用 OAuth2TokenApi.checkAccessToken(token) (WebClient)
    │   ├─ 本地缓存校验结果（1 分钟，Guava Cache）
    │   └─ 解析后将用户信息写入 login-user Header 转发下游
    │
    ├─ Web 层:
    │   TokenAuthenticationFilter (OncePerRequestFilter)
    │   ├─ 优先从 login-user Header 解析（Gateway 透传场景）
    │   ├─ 其次从 Authorization Header 获取 Token
    │   ├─ 调用 OAuth2TokenApi.checkAccessToken(token) (Feign)
    │   └─ 验证通过后写入 SecurityContext
    │
    └─ 下游服务:
        从 SecurityContext 获取 LoginUser → 业务处理
```

> **说明**: 存在两种 Token 解析路径：一是 Gateway 已解析后通过 `login-user` Header 透传，下游服务无需再次校验 Token（高性能路径）；二是直接到达 Web 层，需要通过 Feign 调用认证服务校验 Token（标准路径）。

### 6.2 双 Token 机制

项目实现了标准的 OAuth2 双 Token 机制：

| Token 类型 | 有效期 | 存储位置 | 用途 |
|-----------|--------|----------|------|
| Access Token | 短期（如 2 小时） | Redis + MySQL | 业务接口认证 |
| Refresh Token | 长期（如 30 天） | 仅 MySQL | 刷新 Access Token |

**Token 校验**: `OAuth2TokenServiceImpl.checkAccessToken()`

```
checkAccessToken(token)
    → 1. 优先从 Redis 查询 Token 信息（高性能）
    → 2. Redis 未命中，从 MySQL 查询
    → 3. 验证 Token 有效性（是否过期、是否已撤销）
    → 4. 验证通过后缓存到 Redis（下次快速校验）
    → 5. 返回 Token 对应的用户信息
```

### 6.3 Gateway 层处理

`platform-gateway` 模块中的 `TokenAuthenticationFilter`（实现 `GlobalFilter`）负责统一认证入口：

- 使用 **Reactive** 编程模型（WebFlux），适配 Gateway 的异步非阻塞架构
- 本地缓存 Token 校验结果（Guava Cache，1 分钟 TTL），减少对认证服务的压力
- 通过 `WebClient` 调用 `OAuth2TokenApi` 校验 Token（非阻塞 HTTP 客户端）
- 解析 Token 后，将用户信息序列化并通过 `login-user` Header 转发给下游微服务
- 下游服务不再重复校验 Token，直接从 Header 恢复 SecurityContext

> **说明**: Gateway 层统一处理认证，下游服务无需关心 Token 解析逻辑，实现了关注点分离。同时，Gateway 层会过滤掉外部请求中的 `login-user` Header（由客户端清空），防止伪造。

## 7. 安全配置

### 7.1 SecurityProperties

`cn.shutan.platform.framework.security.config.SecurityProperties`

集中管理安全相关的可配置项：

```yaml
platform:
  security:
    # Token 在 HTTP Header 中的名称
    token-header: Authorization
    # Token 在 URL 参数中的名称
    token-parameter: token
    # 是否启用 Mock 模式（开发环境跳过认证）
    mock-enable: false
    # 免登录 URL 列表（不校验 Token）
    permit-all-urls:
      - /login
      - /register
      - /captcha/**
      - /callback/**
```

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `platform.security.token-header` | Authorization | HTTP Header 名称，用于提取 Token |
| `platform.security.token-parameter` | token | URL 参数名，用于提取 Token（WebSocket 等场景） |
| `platform.security.mock-enable` | false | Mock 模式开关，开发环境可开启以跳过认证 |
| `platform.security.permit-all-urls` | [] | 免登录 URL 列表，这些路径不经过 Token 认证 |

### 7.2 模块级安全配置

每个业务模块通过注册 `AuthorizeRequestsCustomizer` Bean，自定义该模块的路径放行规则：

```java
@Bean
public AuthorizeRequestsCustomizer xxModuleCustomizer() {
    return authorizes -> authorizes
        // Swagger 文档
        .requestMatchers("/v3/api-docs/**", "/swagger-ui/**", "/webjars/**").permitAll()
        // 监控端点
        .requestMatchers("/druid/**", "/actuator/**").permitAll()
        // RPC 接口前缀（Feign 调用，内部服务间通信）
        .requestMatchers("/system/rpc/**", "/infra/rpc/**").permitAll()
        // 模块特有的公开接口
        .requestMatchers("/infra/file/get/**").permitAll();
}
```

> **说明**: 各模块的 `AuthorizeRequestsCustomizer` 在 `SecurityConfiguration.java` 中被统一收集并应用到安全配置链中。这种设计使得安全配置是模块自描述的——每个模块声明自己需要放行的路径，而非集中在一处管理。

### 7.3 访问拒绝处理

`cn.shutan.platform.framework.security.core.handler.AccessDeniedHandlerImpl`

当用户已认证但权限不足时，统一返回 403 FORBIDDEN 响应：

```java
// 返回 JSON 格式错误响应
{
    "code": 403,
    "msg": "访问权限不足",
    "data": null
}
```

> **说明**: 项目中区分了两种异常情况：未认证（401 Unauthorized）和已认证但权限不足（403 Forbidden）。`AccessDeniedHandlerImpl` 仅处理后者。未认证的场景由 `AuthenticationEntryPointImpl` 处理，返回 401 响应。

## 8. 安全架构总览

整个 xy-cloud 的安全体系分层设计，从外到内分为以下层次：

```
 ┌──────────────────────────────────────────────────────────────┐
 │                        [Gateway 层]                          │
 │                                                              │
 │  TokenAuthenticationFilter (GlobalFilter)                    │
 │    └─ Token 验证，缓存校验结果，透传 login-user Header         │
 │                                                              │
 │  TenantContextWebFilter                                      │
 │    └─ 从 tenant-id Header 提取租户 ID，设置到上下文            │
 └──────────────────────┬───────────────────────────────────────┘
                        │
 ┌──────────────────────┴───────────────────────────────────────┐
 │                        [Web 层]                              │
 │                                                              │
 │  TenantSecurityWebFilter                                     │
 │    └─ 租户合法性校验 + 防越权检查                              │
 │                                                              │
 │  TokenAuthenticationFilter (OncePerRequestFilter)            │
 │    └─ Token 验证，写入 SecurityContext                        │
 └──────────────────────┬───────────────────────────────────────┘
                        │
 ┌──────────────────────┴───────────────────────────────────────┐
 │                      [AOP 层]                                │
 │                                                              │
 │  @PreAuthorize("@ss.hasPermission(...)")  权限校验           │
 │  @DataPermission                           数据权限控制      │
 │  @TenantIgnore                             忽略租户过滤      │
 │  @Idempotent                               幂等控制          │
 │  @RateLimiter                              限流控制          │
 └──────────────────────┬───────────────────────────────────────┘
                        │
 ┌──────────────────────┴───────────────────────────────────────┐
 │                    [Feign RPC 层]                             │
 │                                                              │
 │  LoginUserRequestInterceptor   透传 LoginUser                │
 │  TenantRequestInterceptor      透传 tenant-id                │
 └──────────────────────┬───────────────────────────────────────┘
                        │
 ┌──────────────────────┴───────────────────────────────────────┐
 │                  [MyBatis Plus 层]                            │
 │                                                              │
 │  TenantDatabaseInterceptor   自动拼接 tenant_id = ?         │
 │  DataPermissionRuleHandler   自动拼接数据权限 WHERE 条件     │
 └──────────────────────────────────────────────────────────────┘
```

> **说明**: 各层职责明确、层层递进。请求先经过 Gateway 层的全局认证和租户提取，再到 Web 层的安全过滤，然后是 AOP 层的声明式控制，最后在 RPC 层和 ORM 层实现数据隔离。这种分层架构既保证了安全性，又提供了灵活的扩展点。

## 9. 代码生成中的安全注解

使用代码生成器创建新模块时，需在 Controller 层添加以下 `@PreAuthorize` 注解：

```java
// CRUD 基础操作
@PreAuthorize("@ss.hasPermission('{module}:{entity}:create')")
@PostMapping("/create")
public CommonResult<Long> create(...) { ... }

@PreAuthorize("@ss.hasPermission('{module}:{entity}:update')")
@PutMapping("/update")
public CommonResult<Boolean> update(...) { ... }

@PreAuthorize("@ss.hasPermission('{module}:{entity}:delete')")
@DeleteMapping("/delete")
public CommonResult<Boolean> delete(...) { ... }

@PreAuthorize("@ss.hasPermission('{module}:{entity}:query')")
@GetMapping("/get")
public CommonResult<?> get(...) { ... }

@PreAuthorize("@ss.hasPermission('{module}:{entity}:query')")
@GetMapping("/page")
public CommonResult<PageResult<?>> page(...) { ... }

@PreAuthorize("@ss.hasPermission('{module}:{entity}:export')")
@GetMapping("/export")
public void export(...) { ... }
```

> **说明**: 模板中的 `{module}` 和 `{entity}` 需替换为实际模块名和实体名（如 `system:user`、`infra:config`）。export 操作通常使用单独的权限字符串，以支持"仅可查看不可导出"的细粒度权限控制。生成后还需在后台管理系统中配置对应的菜单和权限标识，使注解实际生效。
