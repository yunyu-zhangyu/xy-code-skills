# xy-cloud 代码注释规范参考

> **来源**: xy-cloud 项目源码实际风格 + 业界最佳实践
>
> **技术栈**: JDK 8 + Spring Boot 2.7.18 + Swagger (SpringDoc OpenAPI 3)
>
> **说明**: 本文档定义 AI 生成代码时必须遵循的注释规范，涵盖 JavaDoc、行内注释、TODO 标记、日志注释等。目标是让生成的代码"自带文档"，无需人工补充即可阅读。

---

## 1. 注释总原则

| 原则 | 说明 |
|------|------|
| **Public API 必须注释** | Controller、Service 接口、Feign 接口、工具类等所有被外部调用的方法 |
| **实现类选注释** | ServiceImpl 中简单 CRUD 可省略方法注释，复杂业务逻辑必须注释 |
| **字段必须 Swagger** | VO/DO/DTO 字段必须标注 `@Schema`（替代冗余的 `// 字段说明`） |
| **魔数零容忍** | 任何硬编码的数值/字符串必须有注释说明其含义 |
| **注释与代码同行** | 不单独成行的简短说明写在代码右侧 `//` 之后 |

## 2. 代码审查导向注释（核心）

> AI 生成的注释不是为了当"说明书"，而是帮助 reviewer **快速区分哪些是安全样板代码、哪些需要逐行审查**。

### 2.1 注释标记体系

AI 在生成代码时，使用以下四类前缀标记，让 reviewer 一眼定位风险：

| 标记 | 含义 | 对 reviewer 的价值 |
|------|------|-------------------|
| `// →` | **自动生成的样板代码**，无业务风险，字段名映射正确即可 | ✅ 跳过审查 |
| `// === 【核查点】 ===` | **此处需要 reviewer 按实际业务补充逻辑**（如唯一性校验、状态校验、关联数据检查） | 🔴 必看，决定是否遗漏 |
| `// 💡` | **设计决策说明**（如为什么用这个策略、为什么不做某件事） | ℹ️ 可看可不看 |
| `// ⚠️` | **静默失败风险**（代码能编译通过、能运行、但逻辑错了且不报错） | 🔴 必看，xy-cloud 中最隐蔽的坑 |

### 2.2 每次生成文件时 Class-Level JavaDoc 模板

每个 AI 生成的 Java 文件，类注释必须包含 **【AI 生成·核查要点】** 段落：

```java
/**
 * ${businessName} Controller
 *
 * <p>【AI 生成·核查要点】
 * - @PreAuthorize 权限字符串：格式 '{module}:{resource}:{action}'，需确认与后端权限数据一致
 * - @Tag name 呈现在 Swagger UI 分组标题，确认中文命名无歧义
 * - @Validated 类级别 + @Valid 参数级别：新加方法也要标注 @Valid
 *
 * @author 张三
 * @date 2026-06-01
 */
```

各层文件的核查要点参见下表：

| 层 | 核查要点（写入类 JavaDoc） |
|----|-------------------------|
| **Controller** | @PreAuthorize 权限字符串、@Tag 分组命名、@Valid 校验生效、映射路径 `/module/resource` |
| **Service 接口** | 参数/返回值类型是否与需求一致、异常声明是否完整 |
| **ServiceImpl** | @Transactional 事务边界、业务校验是否已补充、BeanUtils 字段映射丢失 |
| **DO** | BaseDO vs TenantBaseDO 继承选择（最高风险）、@TableName 表名、autoResultMap |
| **VO** | @Schema(description) 是否填写、example 是否合理、校验注解(@NotBlank 等)是否完整 |
| **Mapper** | namespace 路径、XML 是否同时创建、参数类型匹配 |

### 2.3 审查友好 vs. 教育型注释的对比

```java
// ❌ 教育型注释（对 reviewer 毫无价值）
// 将请求 VO 转换为 DO 实体
${entity}DO entity = BeanUtils.toBean(reqVO, ${entity}DO.class);

// ✅ 审查友好注释
// → BeanUtils.toBean 按字段名自动映射（字段名不一致时会丢失数据，请审查）
${entity}DO entity = BeanUtils.toBean(reqVO, ${entity}DO.class);


// ❌ 教育型注释
// 校验记录是否存在
if (existing == null) { throw new ServiceException(NOT_FOUND); }

// ✅ 审查友好注释
// === 【核查点】name 唯一性校验：此处已被 AI 补充，请确认是否匹配业务规则 ===
if (existing != null) { throw new ServiceException(NAME_DUPLICATE); }
```

### 2.4 各层典型的核查点

| 方法 | 标准注释模板 |
|------|------------|
| Service.createXxx | `// === 【核查点】是否有唯一性校验？如 name 不能重复 ===` |
| Service.updateXxx | `// === 【核查点】更新前是否需校验记录存在？ ===` |
| Service.deleteXxx | `// === 【核查点】删除前是否需要校验关联数据？ ===` |
| Service.getXxxList | `// → 无分页查询，大数据量时评估是否需要分页` |
| DO.继承 | `// ⚠️ 租户表选错为 BaseDO 会导致租户间数据互相可见（无报错！）` |
| DO.JSON字段 | `// ⚠️ JSON 字段必须配合 @TableName(autoResultMap = true)` |
| Feign调用 | `// → 调用 infra 模块 ConfigApi（超时 3s，熔断降级）` |

## 3. 【AI生成】标记生命周期管理

> AI 生成的代码必须附带可搜索/可批量清理的标记，使得"生成→审查→确认→清理"全链路可追溯。

### 3.1 文件级标记规则

每个 AI 自动生成的 `.java` 文件，在 `package` 声明之前必须插入以下 banner 注释块：

```java
/* =========================================================================
 * 【AI 生成标记】— 首次自动生成，请逐项审查确认。
 *
 * █ 生命周期管理:
 *   1. 【AI生成】标记 = 未审查，随时可删除回退
 *   2. 人工确认无误后 → 搜索替换 "【AI生成】" 为 "" 即可清理
 *   3. 若需修改 → 修改后移除对应方法的 【AI生成】标记
 *
 * █ 可搜索：全局 grep "【AI生成】" 定位所有待审查代码
 * ========================================================================= */
```

### 3.2 各层文件 banner 个性化内容

| 文件层 | banner 中强调的检查要点 |
|--------|----------------------|
| **Controller** | 确认 @PreAuthorize 权限字符串匹配、@Tag 分组命名、@Valid 校验生效 |
| **Service 接口** | 确认方法签名/参数/返回值与需求一致 |
| **ServiceImpl** | 确认 @Transactional 事务边界、业务校验已补充、BeanUtils 字段映射 |
| **DO** | DO 继承选择（BaseDO vs TenantBaseDO）是最高风险点 |
| **VO (ReqVO/RespVO)** | 确认 @Schema(description) 与实际字段含义一致 |

### 3.3 方法级标记规则

每个 AI 自动生成的方法，JavaDoc 第一行必须包含 `【AI生成】` 前缀：

```java
/**
 * 【AI生成】创建${businessName}
 *
 * <p>新增一条${businessName}记录，返回自增主键 ID。</p>
 *
 * @param reqVO 创建请求参数（已在 VO 层通过 @Valid 完成字段校验）
 * @return 新记录 ID，封装在 CommonResult 中
 */
```

> **为什么必须用 `【AI生成】` 前缀？** 因为这个标记：
> - 可被 `grep "【AI生成】"` 全局搜索，精确定位所有待审查代码
> - 可被 `sed -i 's/【AI生成】//g'` 批量清理，确认后一键移除
> - 不会被误匹配（中文方括号 + 中文内容，业务代码中几乎不会出现）

### 3.4 生命周期状态转移

```
┌─────────────────────────────────────────────────────┐
│  阶段 1: AI 生成                                     │
│  → 所有方法标记 【AI生成】                             │
│  → 文件头有 AI 生成 banner                            │
│  → 可 grep 搜索全部待审查代码                          │
└──────────────────┬──────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────┐
│  阶段 2: 开发人员审查                                  │
│  ┌──── 审查通过 ────┐  ┌──── 需修改 ────┐             │
│  │ 移除文件 banner  │  │ 修改对应方法    │             │
│  │ 搜索替换         │  │ 移除该方法标记  │             │
│  │ "【AI生成】"→""  │  │ 保留其他方法标记│             │
│  └─────────────────┘  └────────────────┘             │
│  ┌──── 完全废弃 ────┐                                 │
│  │ 直接删除整个文件  │                                 │
│  │ banner 明确标识了 │                                 │
│  │ 这是 AI 生成     │                                 │
│  └─────────────────┘                                 │
└──────────────────┬──────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────┐
│  阶段 3: 清理完成                                     │
│  → 文件中已无 【AI生成】标记                           │
│  → 代码已通过人工审查，质量可追溯                       │
│  → 后续修改按正常代码规范维护                           │
└─────────────────────────────────────────────────────┘
```

### 3.5 回退策略

| 场景 | 操作 |
|------|------|
| 文件完全错了 | `git rm` 直接删除 — banner 明确标识了 AI 来源 |
| 部分方法错了 | `grep -n "【AI生成】" *.java` 找到未审查方法，针对性修改 |
| 整体 OK 但要改一个方法 | 修改该方法逻辑 + 移除该方法的 `【AI生成】` 标记 |
| 全部确认无误 | `grep -rl "【AI生成】" *.java \| xargs sed -i 's/【AI生成】//g'` 批量清理 |

## 4. 类级别注释（JavaDoc）

### 4.1 模板格式

```java
/**
 * {业务中文名} {类职责一句话}
 *
 * @author {作者}
 * @date {创建日期 yyyy-MM-dd}
 */
```

### 2.2 各层类注释规范

| 层 | 类注释模板 | 示例 |
|----|----------|------|
| **Controller** | `{businessName} Controller` | `/** 能源价格 Controller */` |
| **Service 接口** | `{businessName} Service 接口` | `/** 能源价格 Service 接口 */` |
| **ServiceImpl** | `{businessName} Service 实现类` | `/** 能源价格 Service 实现类 */` |
| **Mapper** | `{businessName} Mapper 接口` | `/** 能源价格 Mapper 接口 */` |
| **DO** | `{businessName} DO` | `/** 能源价格 DO */` |
| **VO (ReqVO/RespVO)** | `管理后台 - {businessName} {用途} VO` | `/** 管理后台 - 能源价格新增/修改 Request VO */` |
| **Feign 接口** | `{businessName} Feign 接口` | `/** 能源价格 Feign 接口 */` |
| **DTO** | `{businessName} {用途} DTO` | `/** 能源价格响应 DTO */` |
| **Config** | `{businessName} 配置类` | `/** 多租户 配置类 */` |
| **Job** | `{businessName} 定时任务` | `/** 能源价格统计 定时任务 */` |
| **MQ 消费者** | `{businessName} 消息消费者` | `/** 能源价格变更 消息消费者 */` |
| **Convert** | `{businessName} 对象转换器` | `/** 能源价格 对象转换器 */` |
| **Enum** | `{业务名} 枚举` | `/** 能源价格状态 枚举 */` |
| **测试类** | `{ClassName} 的单元测试类` | `/** EnergyPriceServiceImpl 的单元测试类 */` |

### 2.3 模板中的必填占位符

AI 填充模板时，必须将 `${businessName}` `${author}` `${date}` 替换为实际值：

```java
// ❌ 错误：保留模板占位符
/**
 * ${businessName} Controller
 *
 * @author ${author}
 * @date ${date}
 */

// ✅ 正确：替换为实际内容
/**
 * 能源价格 Controller
 *
 * @author 张三
 * @date 2026-06-01
 */
```

## 3. 方法级别注释

### 3.1 Controller 方法

Controller 方法使用 `@Operation` 注解替代传统 JavaDoc，因为 Swagger UI 直接渲染此注解：

```java
@GetMapping("/page")
@Operation(summary = "获得能源价格分页", description = "支持按名称模糊搜索、按状态筛选、按创建时间范围查询")
@PreAuthorize("@ss.hasPermission('eos:energyprice:query')")
public CommonResult<PageResult<EnergyPriceRespVO>> getEnergyPricePage(
        @Valid EnergyPricePageReqVO pageReqVO) {
    // ...
}
```

| @Operation 属性 | 必填 | 说明 |
|---------------|------|------|
| `summary` | **是** | 一句话描述方法功能（Swagger UI 显示为接口标题） |
| `description` | 推荐 | 详细说明，包括参数组合逻辑、边界行为 |
| `deprecated` | 按需 | `@Deprecated` 注解配合，标记已弃用接口并说明替代方案 |

### 3.2 Service 接口方法

JavaDoc 必须说明：**功能描述、参数含义、返回值意义、异常类型及触发条件**。

```java
/**
 * 创建能源价格
 *
 * @param reqVO 创建请求参数（非空校验已在 VO 层完成）
 * @return 新创建的能源价格 ID
 */
Long createEnergyPrice(@Valid EnergyPriceSaveReqVO reqVO);

/**
 * 更新能源价格
 * <p>
 * 只更新传入的非 null 字段（由 BeanUtils.copyProperties 行为决定）
 * 若传入的 id 不存在，抛出 ENERGY_PRICE_NOT_FOUND 异常
 *
 * @param reqVO 更新请求参数（必须包含 id）
 * @throws ServiceException 如果指定 id 的能源价格不存在
 */
void updateEnergyPrice(@Valid EnergyPriceSaveReqVO reqVO);

/**
 * 删除能源价格（逻辑删除）
 * <p>
 * system 模块使用逻辑删除（deleted=1），eos 模块视业务决定使用物理删除或逻辑删除。
 *
 * @param id 能源价格编号
 * @throws ServiceException 如果指定 id 的能源价格不存在
 */
void deleteEnergyPrice(Long id);

/**
 * 根据编号获得能源价格
 *
 * @param id 能源价格编号
 * @return 能源价格 DO，不存在时返回 null（不做非空断言）
 */
EnergyPriceDO getEnergyPrice(Long id);

/**
 * 获得能源价格列表
 *
 * @param reqVO 查询参数（支持按名称模糊搜索、按状态筛选）
 * @return 能源价格列表，无结果时返回空列表（非 null）
 */
List<EnergyPriceDO> getEnergyPriceList(EnergyPriceListReqVO reqVO);

/**
 * 获得能源价格分页
 *
 * @param pageReqVO 分页查询参数
 * @return 分页结果，无数据时 total=0, list 为空列表
 */
PageResult<EnergyPriceDO> getEnergyPricePage(EnergyPricePageReqVO pageReqVO);
```

### 3.3 ServiceImpl 方法注释原则

| 场景 | 注释要求 |
|------|---------|
| 简单的 CRUD 委托（直接调 Mapper） | 可省略 JavaDoc（接口上已有） |
| 含业务校验逻辑 | 必须在逻辑块前加行内注释 |
| 含外部服务调用（Feign/MQ） | 必须在调用前注释说明依赖 |
| 含复杂条件分支（if-else 超过 3 层） | 每层分支必须注释说明业务含义 |
| 含事务策略（@Transactional 传播行为） | 在方法上方注释说明事务边界 |

```java
@Override
@Transactional(rollbackFor = Exception.class)
public Long createEnergyPrice(EnergyPriceSaveReqVO reqVO) {
    // 1. 校验：名称不能重复
    EnergyPriceDO existing = energyPriceMapper.selectByName(reqVO.getName());
    if (existing != null) {
        throw new ServiceException(ENERGY_PRICE_NAME_DUPLICATE);
    }

    // 2. 转换并保存
    EnergyPriceDO entity = BeanUtils.toBean(reqVO, EnergyPriceDO.class);
    energyPriceMapper.insert(entity);

    // 3. 发布创建事件（异步 MQ 通知其他模块）
    mqProducer.sendEnergyPriceCreatedEvent(entity.getId());

    return entity.getId();
}
```

### 3.4 Mapper 接口方法

QueryWrapper 中的 lambda 条件必须注释说明业务含义：

```java
default PageResult<EnergyPriceDO> selectPage(EnergyPricePageReqVO reqVO) {
    return selectPage(reqVO, new LambdaQueryWrapperX<EnergyPriceDO>()
            .likeIfPresent(EnergyPriceDO::getName, reqVO.getName())           // 名称模糊搜索
            .eqIfPresent(EnergyPriceDO::getStatus, reqVO.getStatus())         // 状态筛选
            .betweenIfPresent(EnergyPriceDO::getCreateTime, reqVO.getCreateTime()) // 创建时间范围
            .eqIfPresent(EnergyPriceDO::getTenantId, TenantUtils.getTenantId())    // 租户过滤（多租户模块）
            .orderByDesc(EnergyPriceDO::getId));  // 默认按创建顺序倒序
}
```

## 4. 字段级别注释

### 4.1 VO/DTO 字段

**必须使用 `@Schema` 注解**，不要用 `//` 行内注释代替：

```java
// ❌ 错误：使用行内注释代替 Swagger 注解
private String name; // 价格名称

// ✅ 正确：使用 @Schema 描述字段
@Schema(description = "价格名称", requiredMode = Schema.RequiredMode.REQUIRED, example = "峰时电价")
private String name;
```

| @Schema 属性 | 必填 | 说明 |
|-------------|------|------|
| `description` | **是** | 字段中文含义，要求不含糊、不自解释（如"状态"应写"状态 0=启用 1=停用"） |
| `requiredMode` | 按需 | 必填字段标注 `Schema.RequiredMode.REQUIRED` |
| `example` | **推荐** | 提供合理的示例值，便于 Swagger UI 调试 |
| `hidden` | 按需 | 不在 Swagger UI 显示（如内部字段） |

### 4.2 DO 字段

DO 字段可以使用 `//` 行内注释（因为 DO 不直接暴露给 API 调用方），但**业务关键字段必须有注释**：

```java
// ❌ 差：初学者无法理解
private Integer status;

// ✅ 好：说明枚举含义
private Integer status;  // 0=启用 1=停用 2=冻结

// ✅ 更好：引用枚举类
/**
 * 状态
 * @see CommonStatusEnum
 */
private Integer status;
```

### 4.3 枚举/常量字段

所有枚举必须附加注释说明：

```java
// ❌ 差：纯靠猜
SUCCESS,
FAIL;

// ✅ 好：一目了然
SUCCESS("操作成功"),
FAIL("操作失败"),
```

## 5. 行内注释规范

### 5.1 什么时候必须加

| 场景 | 示例 |
|------|------|
| 魔数/魔字符串 | `pageSize = 1000; // 单次导出最大行数限制` |
| 复杂条件分支 | `if (user.getType() == 1) { // 管理员跳过部门权限校验` |
| 非直观的业务规则 | `// 仅在工作日（周一至周五）触发定时任务` |
| 特殊 null 处理 | `// 为 null 时不覆盖已有字段（仅更新传入的非 null 字段）` |
| 临时兼容逻辑 | `// TODO: v2.0 移除该兼容逻辑` |
| 线程安全说明 | `// 使用 ConcurrentHashMap 确保可见性` |

### 5.2 格式要求

```java
// 行内注释：双斜杠 + 一个空格 + 中文（或英文）句子
// 句末可省略句号，多个句子必须加句号

// 如果注释与代码同行：代码与 // 之间至少两个空格
private Integer status;  // 0=启用 1=停用

// 如果注释独占一行：上方空一行（第一个方法内可不空），与下方代码无空行
// 校验部门是否存在
DeptDO dept = deptMapper.selectById(deptId);
```

## 6. TODO / FIXME / HACK 标记

### 6.1 标记格式

```java
// TODO: {负责人} {截止日期} {需要做的事}
// FIXME: {负责人} {存在的缺陷}
// HACK: {负责人} {为什么需要这样绕}
// XXX: {负责人} {虽然能工作但需要警惕的问题}
```

### 6.2 示例


## 7. 特殊场景注释

### 7.1 @SuppressWarnings 必须写原因

```java
// ✅ 正确：说明为什么忽略警告
@SuppressWarnings("unchecked")  // MapStruct 生成代码，类型安全由 IDE 编译检查保证
@SuppressWarnings("rawtypes")   // 泛型擦除是兼容旧版 SDK 需要
```

### 7.2 测试方法注释

测试方法建议使用**三段式注释**（Arrange-Act-Assert）：

```java
@Test
public void testCreateEnergyPrice_success() {
    // 准备参数（Arrange）
    EnergyPriceSaveReqVO reqVO = randomPojo(EnergyPriceSaveReqVO.class);
    // mock 外部依赖
    when(externalService.validate(any())).thenReturn(true);

    // 调用（Act）
    Long id = energyPriceService.createEnergyPrice(reqVO);

    // 断言（Assert）
    assertNotNull(id);
    EnergyPriceDO entity = energyPriceMapper.selectById(id);
    assertPojoEquals(reqVO, entity, "id", "createTime", "updateTime");
}
```

### 7.3 复杂 SQL 注释

MPJLambdaWrapper 中的 Join 条件必须逐行注释：

```java
default PageResult<EnergyPriceRespVO> selectJoinPage(EnergyPricePageReqVO reqVO) {
    return selectJoinPage(reqVO, EnergyPriceRespVO.class,
            new MPJLambdaWrapperX<EnergyPriceDO>()
                .selectAll(EnergyPriceDO.class)                           // 主表所有字段
                .selectAs(PriceCategoryDO::getName, EnergyPriceRespVO::getCategoryName) // 关联查询分类名称
                .leftJoin(PriceCategoryDO.class, PriceCategoryDO::getId, EnergyPriceDO::getCategoryId) // LEFT JOIN 分类表
                .likeIfPresent(EnergyPriceDO::getName, reqVO.getName())   // 价格名称模糊搜索
                .eqIfPresent(EnergyPriceDO::getStatus, reqVO.getStatus()) // 状态筛选
                .orderByDesc(EnergyPriceDO::getId));                      // 按 ID 倒序
}
```

### 7.4 模块间 Feign 调用注释

调用跨模块 Feign 接口时，必须注释**被调用的模块名 + 超时配置**：

```java
// 调用 infra 模块的 ConfigApi 获取文件上传大小限制（超时 3s，熔断降级返回默认值 10MB）
String maxSizeStr = configApi.getConfigValueByKey("file.max-size");
```

## 8. 覆写方法注释原则

| 场景 | 注释规则 |
|------|---------|
| 覆写父类方法、逻辑完全相同 | **不写 JavaDoc**，父类已有 |
| 覆写方法、扩展了父类逻辑 | 写 JavaDoc 说明**额外行为** |
| 实现接口方法、逻辑较复杂 | 写 JavaDoc 或行内注释说明实现细节 |

```java
@Override
// 此处覆写：在父类逻辑基础上增加了缓存清理操作
public void updateEnergyPrice(EnergyPriceSaveReqVO reqVO) {
    super.updateEnergyPrice(reqVO);
    // 清理该能源价格的本地缓存
    CacheUtils.invalidate("energy_price_" + reqVO.getId());
}
```

## 9. AI 生成注释检查清单

AI 完成代码生成后，对**每个文件**逐一检查以下 8 项：

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 类 JavaDoc | 类名上方有 `/** ... */`，含中文业务描述 |
| 2 | 模板占位符替换 | `${businessName}` `${author}` `${date}` 已替换为实际值 |
| 3 | Controller @Operation | 每个公开方法有 `@Operation(summary = "...")` |
| 4 | VO 字段 @Schema | 每个字段有 `@Schema(description = "...")` 含 example |
| 5 | Service 接口 JavaDoc | 每个方法有 `@param` 和 `@return` 说明 |
| 6 | 魔数注释 | 所有硬编码数字/字符串有注释 |
| 7 | QueryWrapper 条件注释 | 每个 `.eqIfPresent()` / `.likeIfPresent()` 有注释 |
| 8 | TODO 真实有效 | 无空的 `// TODO: 待实现` 等无意义标记 |
