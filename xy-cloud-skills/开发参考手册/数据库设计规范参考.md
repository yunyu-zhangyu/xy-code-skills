# xy-cloud 数据库设计规范参考

> 来源: xy-cloud 项目源码（`platform-module-system`, `platform-module-infra`, `platform-spring-boot-starter-mybatis`）
> 基于 MyBatis-Plus 3.5.7 + H2/MySQL，根包 `cn.shutan.platform`

---

## 1. 表命名规范

### 1.1 命名格式

表名遵循 **全小写蛇形命名（snake_case）**，格式为:

```
{module}_{entity}
```

| 模块前缀 | 对应模块 | 示例 |
|---------|---------|------|
| `system_` | platform-module-system | system_dept, system_users, system_role |
| `infra_` | platform-module-infra | infra_config, infra_file, infra_api_access_log |

### 1.2 命名规则

- **全部小写**，单词之间用下划线分隔
- **前缀表示模块归属**: `system_` 表示系统模块，`infra_` 表示基础设施模块
- **实体名通常用单数**: `dept`, `role`, `menu`, `config`
- **少数表用复数**: `system_users` —— 因为 `user` 是 SQL Server 关键字，避免冲突
- **关联表用实体名组合**: `system_role_menu`, `system_user_role`, `system_user_post`

### 1.3 `@TableName` 注解

所有 DO 类必须标注 `@TableName` 指定表名:

```java
@TableName("system_dept")
@TableName(value = "system_users", autoResultMap = true)    // JSON 字段时需 autoResultMap
@TableName(value = "system_oauth2_client", autoResultMap = true)
```

> **说明**: `autoResultMap = true` 在使用了 `JacksonTypeHandler`（JSON 类型处理器）时**必须开启**，否则 JSON 字段无法正确映射。

## 2. DO 类继承体系

项目采用**两层继承体系**: `BaseDO`（公共基类）和 `TenantBaseDO`（多租户扩展）。

### 2.1 BaseDO（所有 DO 基类）

**全限定名**: `cn.shutan.platform.framework.mybatis.core.dataobject.BaseDO`

BaseDO 为所有数据库实体提供公共字段:

| 字段 | 类型 | 注解 | 填充策略 | 说明 |
|------|------|------|---------|------|
| id | Long | `@TableId` | `ASSIGN_ID` | 雪花算法分布式主键 |
| createTime | LocalDateTime | `@TableField(fill = INSERT)` | 插入时自动填充 | 记录创建时间 |
| updateTime | LocalDateTime | `@TableField(fill = INSERT_UPDATE)` | 更新时自动填充 | 记录更新时间 |
| creator | String | `@TableField(fill = INSERT)` | 插入时自动填充 | 创建者（当前用户 ID） |
| updater | String | `@TableField(fill = INSERT_UPDATE)` | 更新时自动填充 | 更新者（当前用户 ID） |
| deleted | Boolean | `@TableLogic` | — | 逻辑删除标记 |

除 MyBatis-Plus 注解外，BaseDO 还实现了:
- `Serializable` —— 支持序列化
- `TransPojo`（Easy-Trans 字典翻译接口） —— 支持字段值自动翻译为字典标签

### 2.2 TenantBaseDO（多租户扩展）

**全限定名**: `cn.shutan.platform.framework.tenant.core.db.TenantBaseDO`

继承 BaseDO，增加租户 ID 字段:

| 字段 | 类型 | 说明 |
|------|------|------|
| tenantId | Long | 租户 ID |

DDL 定义中默认值为 `'0'`:

```sql
"tenant_id" bigint not null default '0'
```

### 2.3 继承选择原则

```
场景                          → 继承
───────────────────────────────────
系统级表（字典/菜单/岗位/配置） → BaseDO
租户级表（部门/用户/角色）     → TenantBaseDO
```

**判断标准**: 该表的数据是否需要按租户隔离。如果是每个租户独立的数据（如部门、用户、角色），继承 `TenantBaseDO`；如果是系统全局共享的数据（如字典、配置、菜单），继承 `BaseDO`。

> ⚠️ **选错后果（高危）**：租户表（如部门/用户）错误继承 `BaseDO` 时，`TenantDatabaseInterceptor` 不会自动拼接 `WHERE tenant_id = ?` 过滤条件。结果是**租户 A 可以看到租户 B 的全部数据**——无报错、无异常、日志无痕迹。单租户环境测试无法发现此问题，只有多租户并发时才会暴露。这是项目中最危险的数据安全问题之一。

### 2.5 eos / demand 模块 DO 体系（例外）

eos 模块和 demand-response 模块采用与 system/infra 不同的租户处理方式：

**所有 DO 继承 `BaseDO`（非 `TenantBaseDO`）**。虽然数据库表中包含 `tenant_id` 字段，但租户过滤不在框架层自动完成，而是在 Mapper XML 中**手动拼接** `tenant_id = #{tenantId}` 条件。

这与 system/infra 模块的根本区别：

| | system / infra | eos / demand |
|--|---------------|-------------|
| DO 父类 | `TenantBaseDO` | `BaseDO` |
| 租户过滤方式 | 框架自动（`TenantDatabaseInterceptor`） | XML 手动拼接 |
| tenant_id 字段 | 父类自带 | 在 DO 中手动声明 |

> **为什么 eos 不走自动租户过滤**：eos 作为能效运营系统，部分业务查询需要跨租户聚合分析（如总能耗统计），自动租户过滤会限制这类查询。因此 eos 采用更灵活的手动控制方式，让开发者自行决定哪些查询需要带租户条件。

## 3. 主键设计

### 3.1 主键字段类型

所有 DO 统一使用 **Long 类型雪花算法 ID**:

```java
@TableId  // 默认使用 IdType.ASSIGN_ID（雪花算法）
private Long id;
```

> **为什么不用自增主键?**: 分布式环境下自增 ID 会产生主键冲突，雪花算法在分布式场景下可生成全局唯一 ID，且保持有序性。

### 3.2 多数据库序列兼容（KeySequence）

```java
@KeySequence("system_dept_seq")
@KeySequence("system_users_seq")
```

**命名模式**: `{table_name}_seq`

> **说明**: `@KeySequence` 用于支持需要使用序列生成主键的数据库，包括 **Oracle、PostgreSQL、H2、Kingbase（人大金仓）、DM（达梦）** 等。MySQL 不使用序列，但保留注解以保持代码的可移植性。

### 3.3 DDL 主键定义

```sql
"id" bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY
```

> **说明**: 使用 `GENERATED BY DEFAULT AS IDENTITY` 语法是标准 SQL 方式，兼容 H2 和 MySQL。当显式插入 ID 值时，不会与自动生成冲突。

## 4. 通用字段约定

**每个表都必须包含**由 BaseDO 提供的以下公共字段:

| 字段名 | 类型 | DDL | 说明 |
|--------|------|-----|------|
| create_time | timestamp | NOT NULL | 创建时间，插入时自动填充 |
| update_time | timestamp | NOT NULL | 更新时间，插入和更新时自动填充 |
| creator | varchar(64) | DEFAULT '' | 创建者，自动填充当前登录用户 ID |
| updater | varchar(64) | DEFAULT '' | 更新者，自动填充当前登录用户 ID |
| deleted | bit(1) | NOT NULL DEFAULT FALSE | 逻辑删除标记，0=未删除，1=已删除 |
| tenant_id | bigint | NOT NULL DEFAULT '0' | 租户 ID（仅 TenantBaseDO 继承的表需要） |

> **最佳实践**: DDL 中 `deleted` 定义为 `NOT NULL DEFAULT FALSE`，避免 NULL 值参与逻辑判断导致意外结果。

## 5. 自动填充机制

### 5.1 DefaultDBFieldHandler

**全限定名**: `cn.shutan.platform.framework.mybatis.core.handler.DefaultDBFieldHandler`

该类实现了 MyBatis-Plus 的 `MetaObjectHandler` 接口，并注册为 Spring Bean，负责自动填充 BaseDO 中的审计字段。

**插入时填充**（当字段值为 null 时）:

| 字段 | 填充值 |
|------|--------|
| createTime | `LocalDateTime.now()` |
| updateTime | `LocalDateTime.now()` |
| creator | `LoginUser.getId().toString()`（当前登录用户 ID） |
| updater | `LoginUser.getId().toString()`（当前登录用户 ID） |

**更新时填充**（当字段值为 null 时）:

| 字段 | 填充值 |
|------|--------|
| updateTime | `LocalDateTime.now()` |
| updater | `LoginUser.getId().toString()`（当前登录用户 ID） |

> **关键设计**: 仅当对象是 `BaseDO` 实例时触发填充；仅当字段值为 **null** 时才填充——用户显式设置的值**优先保留**，不会被覆盖。

### 5.2 `@TableField` 填充策略

| 策略 | 说明 |
|------|------|
| `FieldFill.INSERT` | 仅插入时自动填充 |
| `FieldFill.INSERT_UPDATE` | 插入和更新时自动填充 |
| `FieldFill.UPDATE` | 仅更新时自动填充 |
| `FieldFill.DEFAULT` | 默认不自动填充 |

## 6. 字段注解 `@TableField` 使用模式

### 6.1 注解模式速查

| 用途 | 示例 | 说明 |
|------|------|------|
| 自动填充 | `@TableField(fill = FieldFill.INSERT)` | 配合 `MetaObjectHandler` 实现审计字段自动赋值 |
| JSON 类型处理器 | `@TableField(typeHandler = JacksonTypeHandler.class)` | 将 List/Set/Map 序列化为 JSON 字符串存入单字段 |
| 非数据库字段 | `@TableField(exist = false)` | 标记为 Java 内存字段，不参与持久化 |
| 强制更新 | `@TableField(updateStrategy = FieldStrategy.ALWAYS)` | 字段值为 null 时仍然执行 UPDATE 操作 |
| 加密处理器 | `@TableField(typeHandler = EncryptTypeHandler.class)` | 敏感字段（如数据库密码）自动加解密 |
| 指定 JDBC 类型 | `@TableField(jdbcType = JdbcType.VARCHAR)` | 解决部分数据库类型推断不准确的问题 |

### 6.2 JacksonTypeHandler 适用场景

`JacksonTypeHandler` 用于将 **复杂 Java 类型序列化为 JSON 字符串** 存储在单个数据库字段中。以下为项目中的实际使用场景:

| DO 类 | 字段 | Java 类型 | 存储内容 |
|-------|------|-----------|---------|
| AdminUserDO | postIds | `Set<Long>` | 用户关联的岗位 ID 集合 |
| TenantPackageDO | menuIds | `Set<Long>` | 租户套餐可用的菜单 ID 集合 |
| OAuth2AccessTokenDO | scopes | `List<String>` | Token 的权限范围列表 |
| SmsTemplateDO | params | `List<String>` | 短信模板参数名列表 |
| SmsLogDO | templateParams | `Map<String, Object>` | 短信模板实际参数键值对 |
| RoleDO | dataScopeDeptIds | `Set<Long>` | 数据权限可见的部门 ID 集合 |
| FileConfigDO | config | `Map<String, Object>` | 文件存储配置（如 OSS 配置） |

**必须配套开启 autoResultMap**:

```java
@TableName(value = "system_users", autoResultMap = true)
```

> **说明**: `autoResultMap = true` 告诉 MyBatis-Plus 为当前实体生成 `ResultMap`，使 `typeHandler` 在查询时能正确反序列化 JSON 字符串为 Java 对象。
>
> ⚠️ **未开启的后果**：JacksonTypeHandler 不会生效，JSON 字段查询返回值为 **null**（静默失败，无任何异常或错误日志），极易被当作"数据未写入"而误导排查方向。项目已有踩坑案例：`AdminUserDO.postIds` 曾因忘记开启而功能异常。

## 7. 逻辑删除

### 7.1 配置方式

在 BaseDO 的 `deleted` 字段上标注 `@TableLogic`:

```java
@TableLogic
private Boolean deleted;
```

DDL 定义:

```sql
"deleted" bit NOT NULL DEFAULT FALSE
```

MyBatis-Plus 全局配置:

```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-value: 1        # 已删除
      logic-not-delete-value: 0     # 未删除
```

### 7.2 查询自动过滤

一旦配置了 `@TableLogic`，**所有 Mapper 查询将自动追加 `WHERE deleted = 0`** 条件，包括 `selectList`、`selectOne`、`selectPage` 等 MyBatis-Plus 内置方法。

如果使用自定义 SQL 编写关联查询，也需要**显式加上删除条件**:

```sql
AND u.deleted = 0
```

> **说明**: 逻辑删除是软删除，数据并未真正从数据库移除，仅标记为已删除。这样做的优势是可以恢复数据、保留历史记录、维护数据完整性。

## 8. 多租户设计

### 8.1 字段定义

多租户字段定义在 `TenantBaseDO` 中:

```java
private Long tenantId;
```

每个继承 `TenantBaseDO` 的表都必须包含 `tenant_id` 字段:

```sql
"tenant_id" bigint NOT NULL DEFAULT '0'
```

### 8.2 自动过滤机制

`TenantDatabaseInterceptor` 实现了 MyBatis-Plus 的 `TenantLineHandler` 接口，**自动为所有查询拼接 `AND tenant_id = ?`** 条件。开发者无需在每个查询中手动指定租户条件。

### 8.3 忽略租户过滤的表

某些系统级表不需要租户过滤，通过配置文件指定:

```yaml
platform:
  tenant:
    ignore-tables:
      - system_tenant
      - system_dict_type
      - system_dict_data
      - infra_config
      # ... 其他系统级配置表
```

### 8.4 编程式忽略

在某些特殊场景下（如管理员后台跨租户操作），需要临时跳过租户过滤:

**注解方式**:
```java
@TenantIgnore  // 标注在 Service 方法上，该方法内所有查询不追加租户条件
public void someMethod() { ... }
```

**编程方式**:
```java
TenantUtils.executeIgnore(() -> {
    // 该代码块内的所有数据库操作不追加租户条件
    return someService.listAll();
});
```

> **说明**: 两种方式底层共享同一个 `ThreadLocal` 标记，`@TenantIgnore` 适合声明式场景，`TenantUtils.executeIgnore` 适合编程式场景。

## 9. 数据库设计示例

### 9.1 系统模块表（继承 TenantBaseDO）

以 `system_dept`（部门表）为例:

```sql
CREATE TABLE "system_dept" (
    "id"          bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    "name"        varchar(30) NOT NULL,
    "parent_id"   bigint,
    "status"      int NOT NULL,
    "sort"        int NOT NULL,
    "create_time" timestamp   NOT NULL,
    "update_time" timestamp   NOT NULL,
    "creator"     varchar(64) DEFAULT '',
    "updater"     varchar(64) DEFAULT '',
    "deleted"     bit         NOT NULL DEFAULT FALSE,
    "tenant_id"   bigint      NOT NULL DEFAULT '0'
);
```

> **说明**: 这是一个典型的租户级表，包含 BaseDO 全部公共字段 + `tenant_id`。业务字段 `name`、`parent_id`、`status`、`sort` 按业务需求定义。

### 9.2 基础模块表（继承 BaseDO）

以 `infra_config`（配置表）为例:

```sql
CREATE TABLE "infra_config" (
    "id"          bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    "category"    varchar(64)  NOT NULL,
    "type"        int          NOT NULL,
    "name"        varchar(100) NOT NULL,
    "value"       varchar(500) NOT NULL,
    "visible"     bit          NOT NULL,
    "remark"      varchar(500),
    "create_time" timestamp    NOT NULL,
    "update_time" timestamp    NOT NULL,
    "creator"     varchar(64)  DEFAULT '',
    "updater"     varchar(64)  DEFAULT '',
    "deleted"     bit          NOT NULL DEFAULT FALSE
);
```

> **说明**: 这是一个系统级表，包含 BaseDO 公共字段但没有 `tenant_id`。其数据在所有租户间共享。

### 9.3 DDL 模板

**新建继承 BaseDO 的表** —— DDL 模板:

```sql
CREATE TABLE "{module}_{entity}" (
    "id"          bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    -- 业务字段在此定义 --
    "create_time" timestamp   NOT NULL,
    "update_time" timestamp   NOT NULL,
    "creator"     varchar(64) DEFAULT '',
    "updater"     varchar(64) DEFAULT '',
    "deleted"     bit         NOT NULL DEFAULT FALSE
);
```

**新建继承 TenantBaseDO 的表** —— DDL 模板:

```sql
CREATE TABLE "{module}_{entity}" (
    "id"          bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY,
    -- 业务字段在此定义 --
```

## 10. 特殊说明

### 10.1 乐观锁

项目**未使用**乐观锁机制，DO 中不包含 `@Version` 注解，也不启用 MyBatis-Plus 乐观锁插件。在并发场景下建议通过 `select ... for update` 悲观锁或分布式锁来解决。

### 10.2 索引定义

索引通过 **原生 DDL 语句** 定义，不使用 `@TableIndex` 注解:

```sql
CREATE INDEX idx_system_users_dept_id ON system_users(dept_id);
CREATE UNIQUE INDEX idx_system_role_name ON system_role(name, tenant_id);
```

> **说明**: DDL 方式更直观、可控，且不依赖 ORM 框架的索引处理能力。

### 10.3 状态字段类型规则（分模块）

| 模块 | 状态字段 Java 类型 | 数据库类型 | 取值示例 |
|------|-------------------|-----------|---------|
| system / infra | `Integer` | `tinyint(3)` | 0=启用, 1=停用 |
| eos / demand-response | `String` | `varchar(8)` | '01', '0201' |

> ⚠️ **system/infra 模块禁止使用 String 存状态值**。若状态字段为 `String` 类型，`LambdaQueryWrapperX.eqIfPresent()` 遇到 `Integer` 类型的查询参数时会静默跳过该条件（类型不匹配），导致查询返回全量数据。项目实际存在不一致（`AdminUserDO.status=Integer` 但 `DeptDO.status=String`），新实体必须统一使用 `Integer`。

### 10.4 枚举字段

枚举字段使用 **`@InEnum` 自定义校验注解** 进行参数校验，而非 MyBatis-Plus 的通用枚举类型处理器:

```java
@InEnum(value = CommonStatusEnum.class, message = "状态必须在 {value} 中")
private Integer status;
```

> **说明**: 数据库中存储的是枚举的 `int` 值，通过 Bean Validation 在校验层确保传入值合法，避免了枚举序列化/反序列化的复杂性。

### 10.5 加密存储

敏感字段（如数据源密码、第三方密钥）使用 `EncryptTypeHandler` 处理器实现透明加解密:

```java
@TableField(typeHandler = EncryptTypeHandler.class)
private String password;
```

> **说明**: 写入数据库时自动加密，读取时自动解密，业务代码无需感知加解密细节。

| 主键 | Long + 雪花算法 | `@TableId` 默认策略 |
| 逻辑删除 | `@TableLogic` | 所有表必须有 deleted 字段 |
| 审计字段 | 由 BaseDO 提供 | create_time/update_time/creator/updater |
| JSON 字段 | JacksonTypeHandler + `autoResultMap = true` | **未开启会导致字段静默返回 null** |
| 多租户 | TenantBaseDO | 自动 SQL 拼接。选错 BaseDO 导致数据泄露 |
| 状态字段(system/infra) | `Integer` + `tinyint(3)` | **禁止**用 String，会导致查询条件静默失效 |
| 状态字段(eos) | `String` + `varchar(8)` | eos 模块例外，用字符串编码 |
| 枚举校验 | `@InEnum` | 数据库存 int（system/infra）/ varchar（eos） |
| 加密存储 | EncryptTypeHandler | 透明加解密 |
